### 2. Описание исходных данных

**OWASP Juice Shop** выбран как эталонный учебный объект: проект входит в линейку флагманских инициатив OWASP, широко используется для практики безопасной разработки и тестирования, поставляется в виде контейнерного образа и воспроизводимых инструкций из README. Это намеренно уязвимое веб-приложение — «витрина» типичных проблем безопасности для обучения без нарушения реальных систем.

По функциональности приложение имитирует интернет-магазин: одностраничный клиент (SPA) взаимодействует с HTTP API сервера на Node.js, пользователи регистрируются, входят в систему, просматривают каталог, управляют корзиной и оформляют заказы; данные размещаются в локальной БД в составе процесса приложения.

### 3. Настройка и развёртывание

**Целевая среда:** Windows 10/11, **Docker Desktop** (режим Linux-контейнеров), доступ в браузере к приложению по **`http://localhost:3000`** (или другому проброшенному порту хоста, например `http://localhost:8080` при `-p 8080:3000`; также `127.0.0.1`). Образ сборки upstream ориентирован на **Node.js 20** в минималистичной среде исполнения (в репозитории используется образ на базе **distroless `nodejs20`** — см. `Dockerfile` в upstream).

**Запуск по README (официальный образ):**

```powershell
docker pull bkimminich/juice-shop
docker run --rm -p 127.0.0.1:3000:3000 bkimminich/juice-shop
```

После старта веб-интерфейс доступен по адресу `http://localhost:3000` (маппинг только на интерфейс loopback хоста).

**Альтернатива (практическое руководство курса):** проброс на все интерфейсы хоста, что удобно для быстрой проверки, но расширяет сетевую доступность с точки зрения хоста:

```powershell
docker run -d -p 3000:3000 --name juice-shop bkimminich/juice-shop
```

**Дополнительно (опционально):** клонирование `https://github.com/juice-shop/juice-shop` и локальная сборка образа из корня репозитория (`docker build`). В upstream присутствует **`docker-compose.test.yml`**, ориентированный на **CI/тесты** (сборка образа приложения из контекста `./`), а не на типовой «ручной» запуск витрины; для учебной инвентаризации достаточно образа `bkimminich/juice-shop` или собственной сборки по `Dockerfile`.

#### Типичные ошибки при развёртывании и способы устранения

1. **Docker Desktop / демон Docker не запущен** — команды `docker pull` / `docker run` завершаются ошибкой подключения к демону. *Решение:* запустить Docker Desktop и дождаться статуса «running», повторить команду.
2. **Порт 3000 занят** на хосте — контейнер не стартует или маппинг порта отклоняется. *Решение:* освободить порт (остановить конфликтующий процесс/контейнер) либо использовать другой порт хоста, например `-p 127.0.0.1:8080:3000` и открывать `http://localhost:8080`.
3. **Устаревшая схема Docker Machine (legacy) на macOS** — обращение не к `localhost`, а к IP виртуальной машины Docker Machine. *Решение:* использовать актуальный Docker Desktop; при legacy — узнать IP через `docker-machine ip` и подставлять его вместо `localhost` в URL.
4. **Неверная привязка `0.0.0.0` vs `127.0.0.1` в `-p`** — `-p 127.0.0.1:3000:3000` ограничивает доступ **к приложению на хосте** интерфейсом loopback (с других машин в LAN сервис не виден); `-p 3000:3000` по умолчанию слушает на всех адресах хоста и может быть доступен из локальной сети. *Решение:* осознанно выбирать вариант под модель угрозы (учебный стенд на одной машине vs доступ соседних узлов).

### 4. Процесс исследования

Инвентаризация поверхности атаки выполнялась по методике сбора **архитектурных и интерфейсных фактов** без попыток эксплуатации уязвимостей и без использования сканеров/эксплойтов — только документация, конфигурация, наблюдение трафика и среды Docker. Использовались следующие источники:

- **README.md** upstream — назначение проекта, варианты запуска (Docker и др.).
- **`Dockerfile`** — базовый образ, версия Node, директива `EXPOSE`, команда запуска.
- **`docker-compose.test.yml`** — контекст CI: как собирается образ приложения, сервисы тестового контура (не обязательный файл для ручного подъёма витрины).
- **`server.ts`** — объявление маршрутов HTTP API и порядок middleware (авторизация, B2B и пр.).
- **`swagger.yml`** (документация B2B API) — контракт внешнего интерфейса `/b2b/v2`.
- **`config/default.yml`**, **`config.schema.yml`** — имена параметров конфигурации, валидация схемы (для учёта секретов и интеграций **по именам**, без значений).
- **Инструменты среды исполнения:** браузер **DevTools → Network** (фактические запросы UI к API), **Application → Storage** (токены/хранилище клиента); в PowerShell — **`docker ps`**, **`docker port`**, **`docker inspect`**, проверка прослушивания порта на хосте.

**Команды PowerShell из практического руководства (фрагмент):**

```powershell
docker pull bkimminich/juice-shop
docker run -d -p 3000:3000 --name juice-shop bkimminich/juice-shop
docker ps
docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Ports}}\t{{.Status}}"
docker port juice-shop
netstat -ano | findstr 3000
docker inspect juice-shop
```

Дополнительно для сопоставления конфигурации с фактами: просмотр вывода `docker inspect` (сеть, переменные окружения, порты, монтирования) и наблюдение за вкладкой Network при типовых действиях в UI (вход, поиск, регистрация, корзина).

### 5. Результаты инвентаризации

#### 5.1. Границы рассмотрения (scope)

1. **В область анализа** включена связка «хост с Docker Desktop → контейнер с Juice Shop → браузер пользователя на той же машине (типовой сценарий лаборатории)».
2. **Внешняя граница** с точки зрения стенда — **HTTP-доступ с хоста** (браузер к опубликованному порту, например 3000); при привязке `-p 127.0.0.1:3000:3000` **широкая Internet-facing экспозиция на уровне хоста отсутствует**, возможен доступ только локально.
3. **Внутренняя граница** — **процесс Node.js внутри одного контейнера**, включая встроенное хранилище данных и статическую отдачу клиентской части.
4. **Вне области** данной работы: физическая безопасность оборудования, политики ЦОД, **глобальная доступность приложения в Интернете** (если явно не публиковать порт наружу), а также цепочки поставок за пределами выбранного образа/репозитория.
5. **Поверхность атаки** понимается как **совокупность точек входа и доверия** (сеть, API, файлы конфигурации, клиентское хранилище), а не как перечень конкретных CVE.
6. **Цель этапа** — **Inventory** (учёт поверхности по OWASP WSTG и Attack Surface Analysis), без минимизации или защитных мер (hardening).

#### 5.2. Состав системы (компоненты)

| Компонент | Назначение / примечание |
|-----------|-------------------------|
| Клиентская часть | SPA на **Angular** (собранные статические ресурсы отдаются приложением) |
| Сервер приложения | **Express** на **Node.js 20** (как в upstream-образе на базе distroless `nodejs20`) |
| Данные | **SQLite**, доступ через ORM **Sequelize** в **том же процессе**, отдельный сетевой сервис БД не выводится |
| Упаковка | **Один контейнер** «всё в одном» для типового сценария Docker |
| Статические ресурсы | HTML/JS/CSS и прочие ассеты внутри образа / процесса |
| Среда исполнения | **Минимальный (distroless) образ**, снижающий поверхность ОС внутри контейнера по сравнению с полноценным дистрибутивом |

#### 5.3. Сетевые точки доступа

| Параметр | Описание |
|----------|----------|
| Протокол / порт | **TCP 3000**, протокол прикладного уровня **HTTP** (витрина и API на одном порту) |
| Публикация на хосте | Зависит от `docker run -p`: **`127.0.0.1:3000:3000`** — только локальный доступ с хоста; **`3000:3000`** — прослушивание на всех интерфейсах хоста (шире с точки зрения LAN) |
| Декларация образа | В `Dockerfile` присутствует **`EXPOSE 3000`** — документированное ожидание порта контейнера |
| Отдельный порт СУБД | **Не публикуется** — SQLite внутри процесса, без внешнего `3306`/`5432` и т.п. |
| Связь «браузер ↔ приложение» | Единая точка — HTTP к корню и REST/B2B путям на опубликованном порту |
| Диагностика на Windows | `netstat -ano | findstr 3000` — какой PID слушает порт; `docker ps` / `docker port` — маппинг контейнера |

#### 5.4. Точки входа (HTTP API) — иллюстративный перечень

Ниже — примеры эндпоинтов с **фактическими путями**, зарегистрированными в **`server.ts`** upstream (номера строк могут слегка отличаться между версиями; при защите рекомендуется сверить с актуальной веткой).

| Действие (логика) | Endpoint | Метод | Аутентификация (ожидаемо) | Данные (без значений секретов) |
|-------------------|----------|-------|---------------------------|--------------------------------|
| Вход в систему | `/rest/user/login` | `POST` | Не требуется (выдача учётных токенов после успешной проверки) | Тело запроса **JSON**: учётные поля (**email**, **password**) — см. `app.post('/rest/user/login', …)` |
| Поиск по каталогу | `/rest/products/search` | `GET` | Не требуется (публичный поиск) | Строка запроса: параметр **`q`** и др. согласно реализации — см. `app.get('/rest/products/search', …)` |
| Регистрация пользователя | `/api/Users` | `POST` | Не требуется (создание учётной записи) | Тело **JSON**: поля регистрации (в т.ч. **email**, **password**, **passwordRepeat** в типовом потоке) — цепочка `app.post('/api/Users', …)` |
| Получение корзины | `/rest/basket/:id` | `GET` | Требуется (**JWT** / сессионная модель приложения): глобальные `security.isAuthorized()` на префиксы `/rest/basket` | Параметр пути **`:id`** (идентификатор корзины) — `app.get('/rest/basket/:id', …)` |
| Добавление позиции в корзину | `/api/BasketItems` | `POST` | Требуется: middleware на `/api/BasketItems` + контекст пользователя | Тело **JSON**: идентификаторы сущностей (**ProductId**, **BasketId** и связанные поля согласно модели) — `app.post('/api/BasketItems', …)` |
| Оформление B2B-заказа (контракт) | `/b2b/v2/orders` | `POST` | Требуется: `app.use('/b2b/v2', security.isAuthorized())` | Тело запроса по **`swagger.yml`** (структура заказа B2B; детали — из спецификации, без раскрытия секретов) — `app.post('/b2b/v2/orders', …)` |

Ссылки на объявления в upstream `server.ts` (ориентир): `app.post('/rest/user/login', …)`; `app.get('/rest/products/search', …)`; `app.post('/api/Users', …)`; `app.get('/rest/basket/:id', …)`; `app.post('/api/BasketItems', …)`; `app.post('/b2b/v2/orders', …)`.

#### 5.5. Доступ, роли, чувствительные данные и конфигурация

1. **Механизм доступа:** после аутентификации приложение использует **JWT** (заголовок **Authorization: Bearer** и/или передача токена в **cookie** — поведение следует уточнять по клиенту и ответам `/rest/user/login`, без фиксации конкретных значений токенов в отчёте).
2. **Роли (концептуально, учебное приложение):** разграничение возможностей для **клиента (customer)**, **администратора (admin)** и **роли учёта (accounting)** — проявляется в защищённых маршрутах (например, эндпоинты истории заказов с проверкой `isAccounting()` в `server.ts`).
3. **Чувствительные данные предметной области:** учётные и профильные записи **пользователей**, **заказы** и **корзины**, платёжные сущности (в т.ч. API, связанные с **картами** — как интерфейсы хранения/обработки платёжных данных в учебной модели).
4. **Конфигурация и интеграции (только имена):** в конфигурации фигурирует, например, **`googleOauth.clientId`** (идентификатор клиента OAuth при интеграции с Google — значения не приводятся).
5. **Криптографические материалы приложения:** ключи для JWT и прочих нужд хранятся как **файлы в каталоге `encryptionkeys/`** внутри приложения (содержимое файлов и ключи в отчёт **не включаются**).
6. **Переопределение конфигурации:** поддерживается шаблон **`NODE_CONFIG`** (и связанная с **`node-config`** схема слияния конфигов) — **имя переменной** фиксируется для инвентаризации; конкретные секреты и пароли в отчёте **не указываются**.

#### 5.6. Границы доверия (trust boundaries)

| Граница | Участники | Что пересекает границу |
|---------|-----------|-------------------------|
| HTTP/API | Браузер (SPA) ↔ сервер Node/Express | Запросы REST/B2B, ответы JSON, статика, заголовки (в т.ч. **Authorization**) |
| Клиентское хранилище | Код SPA ↔ локальное хранилище браузера | **localStorage** / **sessionStorage**, **cookies** — токены, настройки UI (метаданные сессии) |
| Прикладной уровень ↔ файл БД | Процесс приложения ↔ файл SQLite | Операции ORM к файлу базы внутри контейнера |
| Контейнер ↔ хост | Изолированная среда ↔ Windows + Docker | Проброс порта `-p`, файловая система образа, переменные окружения |
| Внешний IdP | Приложение ↔ **Google OAuth** | Потоки OAuth2/OpenID (по конфигурации; значения client secret не раскрываются) |
| Разработка ↔ репозиторий | Разработчик ↔ **GitLab** (курс) / upstream | Исходный код, CI, **`docker-compose.test.yml`**, история изменений |

### Уточнение терминов (для защиты)

**Поверхность атаки (attack surface)** — это не список дыр, а **перечень интерфейсов и точек взаимодействия**, через которые система принимает входные данные, аутентификацию, сетевые запросы и конфигурацию; в рамках настоящей работы она инвентаризируется без оценки эксплуатируемости.

**Уязвимость (vulnerability)** — это **свойство или ошибка реализации**, которое *может* позволить нарушить конфиденциальность, целостность или доступность при определённых условиях; сама по себе инвентаризация эндпоинтов уязвимостью не является.

**Вектор атаки (attack vector)** — это **конкретный способ доставки** вредоносного или нежелательного воздействия к уязвимому компоненту (например, «недоверенные данные в теле POST к API»), тогда как поверхность отвечает на вопрос *где* система принимает такие данные.
