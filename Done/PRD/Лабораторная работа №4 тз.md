## Теоретическая часть

### Зачем нужен SCA на практике

Современные приложения на 70–90% состоят из открытого кода: согласно отчётам Synopsys OSSRA, в среднем коммерческом приложении детектируется 500+ OSS-компонентов, из которых 84% имеют хотя бы одну известную уязвимость, а 48% — High/Critical. Любая уязвимость в сторонней библиотеке автоматически становится уязвимостью вашего приложения: классический пример — Log4Shell ([CVE-2021-44228](https://nvd.nist.gov/vuln/detail/CVE-2021-44228), CVSS 10.0), затронувший миллионы приложений на JVM в декабре 2021 года просто потому, что они использовали `log4j-core 2.0–2.14.1`.

Композиционный анализ — это процесс, который автоматически отвечает на два вопроса:

1. **«Из чего состоит мой продукт?»** — построение перечня программных компонентов (ППК), он же SBOM (Software Bill of Materials).
2. **«Какие из этих компонентов представляют риск?»** — сопоставление компонентов с базами уязвимостей (NVD, OSV, GHSA, БДУ ФСТЭК), лицензий и политик.

### Прямые и транзитивные зависимости

| Тип | Определение | Пример (Python, проект использует `arrow`) |
|---|---|---|
| **Direct (level 1)** | Подключена разработчиком явно: в `requirements.txt`, `package.json`, `pom.xml` | `arrow==1.3.0` |
| **Transitive (level 2+)** | Подключена транзитивно как зависимость зависимости | `python-dateutil==2.9.0.post0` (нужен `arrow`) |

Это деление критично для исправления: транзитивную уязвимость нельзя «починить локально» — вы либо ждёте патч от автора прямой зависимости, либо переходите на альтернативу, либо принудительно поднимаете версию. Любой SCA-инструмент, который не показывает путь до уязвимости, малополезен на практике.

### CPE и PURL: как идентифицируют компоненты

Чтобы сопоставить «у меня в проекте есть библиотека X версии Y» и «в базе уязвимостей есть запись про X@Y» — нужен унифицированный идентификатор. Используются два:

| Свойство | CPE | PURL |
|---|---|---|
| Назначение | Официальная схема NVD | Открытый стандарт ([purl-spec](https://github.com/package-url/purl-spec)) |
| Формат | `cpe:2.3:a:apache:log4j:2.14.1:*:*:*:*:*:*:*` | `pkg:maven/org.apache.logging.log4j/log4j-core@2.14.1` |
| Источник истины | Централизованный словарь NVD | Можно вывести из манифеста самого пакета |
| Покрытие OSS | Слабое (особенно для npm, PyPI) | Хорошее, поддерживает npm/pypi/maven/gem/cargo/go/docker и др. |

В современных SBOM-форматах (CycloneDX, SPDX 2.2+) обязательным де-факто стал PURL. Если в SBOM `purl` пуст или некорректен — сканер не сможет сопоставить компонент с базами уязвимостей, и компонент «провалится» в отчёте.

### SBOM и форматы

**SBOM** — машиночитаемый перечень компонентов ПО с версиями, идентификаторами, лицензиями и взаимосвязями. NTIA в 2021 году зафиксировала [минимальные обязательные элементы SBOM](https://www.ntia.gov/files/ntia/publications/sbom_minimum_elements_report.pdf): supplier, component name, version, dependency relationship, author, timestamp, уникальный идентификатор (PURL/CPE/SWID).

| Формат | Стандарт | Где сильнее | Где используется |
|---|---|---|---|
| **CycloneDX** | OWASP, ECMA-424 | DevSecOps, supply chain, AppSec; есть VEX, ML-BOM, SaaSBOM | Dependency-Track, Trivy, Syft, GitLab |
| **SPDX** | ISO/IEC 5962:2021 | Лицензионная прозрачность, детализация до файлов | Linux Foundation, Yocto, FOSSology |
| **SWID** | ISO/IEC 19770-2 | IT-инвентаризация установленного ПО | Tanium, эндпоинт-агенты |
| **KBOM** | CycloneDX-надстройка | Инвентарь Kubernetes-кластеров | Trivy (`trivy k8s`) |

**В этой лабораторной мы работаем с CycloneDX**, потому что:

- его умеют генерировать Trivy, Syft, cdxgen, cyclonedx-py и все распространённые package-manager плагины;
- Dependency-Track построен вокруг CycloneDX (SPDX тоже принимает, но CycloneDX — нативный);
- поддерживает PURL, что критично для корреляции с базами уязвимостей.

### Архитектура и принцип работы SCA-инструмента

Любой SCA-сканер выполняет три шага:

1. **Inventory** — обход проекта/образа, парсинг манифестов (`package-lock.json`, `poetry.lock`, `go.sum`, `Gemfile.lock`, OS-пакетов через `dpkg`/`rpm`), сборка списка `(name, version, purl)`.
2. **Match** — поиск каждого компонента в базах уязвимостей: NVD/NIST, OSV.dev, GitHub Advisory Database, БДУ ФСТЭК, собственные сервисы вендора.
3. **Report** — для каждого совпадения: CVE, CWE, CVSS-вектор, описание, fix-версия, путь зависимости.

Качество сканера определяется тремя факторами:

- **полнота баз**: Trivy DB агрегирует NVD + OSV + GitHub + vendor advisories — поэтому даёт больше результатов, чем «голый» pip-audit;
- **точность идентификации**: правильный ли PURL построил инструмент — особенно для транзитивных зависимостей;
- **дедупликация и приоритизация**: умеет ли отличать уязвимости указывающих на один источник.

### Manifest-based(на основе манифестов) vs SBOM-based(на основе SBOM) сканирование

| Подход | Как работает | Плюсы | Минусы |
|---|---|---|---|
| **Manifest-based** (pip-audit, npm audit, trivy fs) | Сканер сам парсит исходники/манифесты при каждом запуске | Простой запуск, всегда «свежий» результат | Каждый CI-job повторяет inventory; нет долгосрочного storage; нет VEX |
| **SBOM-based** (Dependency-Track, OWASP DT) | SBOM генерируется один раз, загружается в платформу, та постоянно сопоставляет его с обновляющимися базами | Continuous monitoring: новая CVE → автоматический алерт по уже загруженным SBOM; единая инвентаризация; политики, VEX | Качество = качество SBOM; нужна инфраструктура (DT-сервер) |

Зрелый процесс использует **оба подхода**: manifest-based в pre-merge (быстро, если использовать на постоянку), SBOM-based — обычно на этапе post-build для проведения постоянного мониторинга.

#### Часть 1. Подготовка проекта

**Цель этапа:** убедиться, что проект пригоден для SCA-анализа: содержит lock-файлы манифесты пакетного менеджера, имеет известные уязвимости.

#### Часть 2. Генерация SBOM с помощью Trivy

Необходимо получить корректный SBOM файл в формате CycloneDX. Пока что мы будем использовать локальную проверку. Поэтому подтягиваем  Trivy с официального registry 

**Шаг 2.1.** Сгенерируйте SBOM в формате CycloneDX для файловой системы проекта:

```bash
docker run --rm \
  -v "$(pwd):/src" \
  aquasec/trivy:latest \
  fs \
  --format cyclonedx \
  --output /src/sbom.cdx.json \
  /src
```

Параметры команды:

| Параметр | Что делает |
|---|---|
| `fs` | Подкоманда: сканировать файловую систему (а не образ или репозиторий) |
| `--format cyclonedx` | Формат вывода SBOM. Альтернативы: `spdx-json`, `github`, `table` |
| `--output /src/sbom.cdx.json` | Куда писать результат (внутри контейнера, через volume — на хост) |
| `/src` | Путь сканирования внутри контейнера |

**Шаг 2.2.** Если ваш проект — контейнеризированное приложение, сгенерируйте также SBOM для собранного образа. Это даёт более полную картину: туда попадут не только зависимости приложения, но и OS-пакеты базового образа.

```bash
# Сначала соберите образ:
docker build -t myapp:lab4 .

# SBOM для образа:
docker run --rm \
  -v "$(pwd):/out" \
  -v /var/run/docker.sock:/var/run/docker.sock \
  localhost:5000/trivy:latest \
  image \
  --format cyclonedx \
  --output /out/sbom-image.cdx.json \
  myapp:lab4
```

> **Различие важно.** SBOM из `fs` покажет ~20–50 зависимостей вашего приложения. SBOM из `image` — те же зависимости плюс 150–300 OS-пакетов из базового образа. Уязвимости в OS-пакетах часто доминируют в отчёте, и вы должны уметь объяснить, к чему относится каждая (приложение или базовый образ).

**Шаг 2.3.** Проверьте корректность SBOM:

```bash
# Базовая валидация JSON:
jq empty sbom.cdx.json && echo "OK"

# Проверка формата и обязательных полей:
jq '.bomFormat, .specVersion, .serialNumber' sbom.cdx.json
# Ожидаемый вывод:

# "1.5"   (или "1.6")
# "urn:uuid:..."

# Все ли компоненты имеют PURL?
```

#### Часть 3. Загрузка SBOM в Dependency-Track

Теперь нам нужно загрузить SBOM в Dependency-Track идним из двух способов способами (UI + API), убедиться, что компоненты проиндексированы и обнаружены уязвимости.

**Шаг 3.1. Создание проекта в Dependency-Track через UI.**

1. Откройте `http://10.0.0.30:8080`, войдите.
2. **Projects → Create Project**:
   - **Name:** `lab4-<фамилия>-<стек>` (например, `lab4-ivanov-python`).
   - **Version:** `1.0.0` или укажите git commit hash.
   - **Classifier:** `Application`.
   - **Description:** ссылка на репозиторий, дата.
3. Сохраните и откройте созданный проект. Скопируйте UUID из URL — он пригодится для API: `http://10.0.0.30:8080/projects/<UUID>`.

**Шаг 3.2. Загрузка SBOM через UI.**

1. На странице проекта откройте вкладку **Components**.
2. Нажмите **Upload BOM**, выберите `sbom.cdx.json`, загрузите.
3. Подождите некоторое время — Dependency-Track индексирует компоненты и подтягивает уязвимости из NVD/OSV/GHSA.
4. Обновите страницу. На вкладке **Components** должны появиться все компоненты, на вкладке **Audit Vulnerabilities** — найденные CVE.

**Шаг 3.3. Получение API-ключа для будущей автоматизации.**

1. Перейдите в **Administration → Access Management → Teams**.
2. Создайте team `lab4-ci-<фамилия>` (или используйте существующий) и выдайте permissions:
   - `BOM_UPLOAD`
   - `VIEW_PORTFOLIO`
   - `PROJECT_CREATION_UPLOAD`
3. Сгенерируйте API key и скопируйте его (отображается один раз).

> **Не вставляйте этот ключ в отчёт и не коммитьте в Git.** В отчёте достаточно описать, *где* он хранится (например: «GitLab CI/CD Variable `DEPENDENCYTRACK_API_KEY`, Masked + Protected»).

**Шаг 3.4. Загрузка SBOM через REST API (рекомендовано, так как это пойдет в CI).**

Повторите загрузку через API — это позволит убедиться, что ключ работает, и понять, что именно будет вызывать пайплайн.

```bash
export DEPENDENCYTRACK_API_KEY="<your-api-key>"
export DEPENDENCYTRACK_PROJECT_UUID="<uuid-проекта-из-UI>"

curl -X POST "http://10.0.0.30:8081/api/v1/bom" \
  -H "X-API-Key: $DEPENDENCYTRACK_API_KEY" \
  -F "project=$DEPENDENCYTRACK_PROJECT_UUID" \
  -F "bom=@sbom.cdx.json"
```

Ожидаемый ответ:

```json
{"token":"a4b6c8d0-..."}
```

Токен можно использовать для опроса статуса обработки:

```bash
curl -s -H "X-API-Key: $DEPENDENCYTRACK_API_KEY" \
  "http://10.0.0.30:8081/api/v1/bom/token/<token>"
# {"processing":false}  → обработка завершена
```

**Шаг 4.3. Приоритизация.** Опишите в отчёте, какой подход применили для приоритизации исправлений. Минимум — учтите следующее:

1. **KEV** (есть подтверждённая эксплуатация in-the-wild) → срок ≤24 ч независимо от CVSS.
2. **CVSS ≥ 9.0 + EPSS ≥ 0.5** → срок ≤72 ч.
3. **CVSS ≥ 7.0 + прямая** → плановое обновление в течение спринта.
4. **CVSS < 7.0 + транзитивная + нет KEV/высокого EPSS** → беклог.

#### Что должно быть в отчёте за базовую часть

- Таблицы 1, 2, 3 (заполненные на ваших данных).
- Скриншоты: страница проекта в Dependency-Track с компонентами; вкладка Audit Vulnerabilities с агрегированной статистикой.

- Раздел «Анализ компонентов» с минимум одним полным обоснованием исправления.

### Дополнительное задание №2 (1 балл). Автоматизация в GitLab CI/CD

**Цель:** перевести ручной процесс из базовой части в пайплайн, который при каждом push в master выполняет SBOM-генерацию, загрузку в Dependency-Track и импорт результатов в DefectDojo.

**Шаг Д2.1. Подготовка переменных в GitLab.**

В вашем GitLab-проекте: **Settings → CI/CD → Variables**. Вы можете перенести все переменные в переменные Gitlab либо оставить их в файле variable.yml. Самое главное оставляйте в файле секреты

| Переменная | Значение | Тип |
|---|---|---|
| `DEPENDENCYTRACK_URL` | `http://10.0.0.30:8081` | Variable |
| `DEPENDENCYTRACK_API_KEY` | (ваш ключ) | Variable, Masked |
| `DEPENDENCYTRACK_PROJECT_UUID` | (UUID проекта из базы) | Variable |
| `DEFECTDOJO_URL` | `http://10.0.0.20:8080` | Variable |
| `DEFECTDOJO_TOKEN` | (токен из DefectDojo, Api v2 Key) | Variable, Masked |
| `DEFECTDOJO_PRODUCTID` | (id продукта в DefectDojo) | Variable |

**Шаг Д2.2. Минимальный рабочий `.gitlab-ci.yml`.**

### Стадия `.pre` — инициализация DefectDojo

Необходимо подготовить DefectDojo к текущему запуску пайплайна — создать новый Engagement и связать его с заранее созданным Product.

**Предусловия:**

- В DefectDojo создан Product, известен его идентификатор `DEFECTDOJO_PRODUCTID`.
- В GitLab CI/CD Variables заданы: `DEFECTDOJO_URL`, `DEFECTDOJO_TOKEN`, `DEFECTDOJO_PRODUCTID`.

**Действия стадии:**

- Рассчитать даты начала и окончания Engagement (`today` и `today + период`).
- Отправить `POST`-запрос в DefectDojo API для создания Engagement (тип `CI/CD`) с привязкой к Product и метаданными пайплайна (pipeline id, branch, commit).
- Сохранить `DEFECTDOJO_ENGAGEMENTID` в файл `defectdojo.env`.

**Результат:** в DefectDojo появляется Engagement для текущего pipeline.

### Стадия `pre-build` — SCA зависимостей проекта

- Запустить SCA-сканирование проекта выбранным инструментом.
- Сформировать отчет.
- Сохранить отчет как артефакт GitLab CI для последующей отправки в DefectDojo.
- *(Опционально)* Вывести в логи краткую сводку: количество найденных уязвимостей и критичность.

### Стадия `build` — генерация SBOM

- Выполнить сборку проекта или контейнерного образа (если предусмотрено процессом).
- Сгенерировать SBOM в формате CycloneDX.
- Сохранить SBOM как артефакт GitLab CI.

### Стадия `post-build` — SCA по SBOM (Dependency-Track)

**Предусловие:** в GitLab CI/CD Variables заданы параметры доступа: `DEPENDENCYTRACK_URL`, `DEPENDENCYTRACK_API_KEY`, `DEPENDENCYTRACK_PROJECT_UUID`, `DEFECTDOJO_ENGAGEMENTID`.

**Действия:**

- Автоматически загрузить SBOM в Dependency-Track (через API).
- Дождаться завершения обработки SBOM.
- Передать результаты Dependency-Track в DefectDojo через внутреннюю интеграцию Dependency-Track (в отчете показать настройки и факт появления findings в DefectDojo).

### Стадия `.post` — отправка результатов в DefectDojo

- Импортировать отчет SCA (из стадии `pre-build`) в DefectDojo через API импорта сканов.
- Импортировать результаты Dependency-Track одним из способов:
  - либо через нативную интеграцию **Dependency-Track ↔ DefectDojo** (если она настроена; в отчете нужно показать настройки и факт появления findings в DefectDojo);
  - либо напрямую — импортом отчета / экспортом findings через API DefectDojo.

**Шаг Д2.3. Проверка работоспособности пайплайна.**

1. Закоммитьте `.gitlab-ci.yml`, дождитесь окончания pipeline.
2. На зелёный pipeline:
   - в Dependency-Track в проекте должны быть обновлены компоненты;
   - в DefectDojo должен появиться новый Test/Engagement с findings из SBOM.

**Шаг Д2.4. Что отразить в отчёте:**

- Полный `.gitlab-ci.yml`.
- Скриншот успешного pipeline (Jobs → все зелёные).
- Скриншот Dependency-Track с обновлённой датой импорта SBOM.
- Скриншот DefectDojo с импортированным Engagement и Findings.
