## 1. Цель работы

Получить практический опыт построения процесса **Управления уязвимостями** для компонентов разрабатываемого приложения: научиться выполнять инвентаризацию компонентов, осуществлять ручной поиск публично известных уязвимостей по открытым источникам, оценивать их применимость и критичность, формировать реестр уязвимостей и вести его в системе управления уязвимостями (DefectDojo).

В рамках данной лабораторной работы студенты:

- проверят **применимость** уязвимости к конкретному стенду;
- корректно формулировать **сценарий эксплуатации** и **рекомендацию по устранению**;

## 3. Теоретическая часть

### 3.1. Зачем нужен процесс управления уязвимостями

За последние годы поток публикуемых уязвимостей растёт практически экспоненциально: в 2024 г. публиковалось ~2 000 CVE в месяц, и тренд сохраняется. При этом одна и та же уязвимость в разных источниках может оцениваться по-разному, что ломает приоритеты исправления. В таких условиях команда разработки не может «фиксить всё подряд» — нужен системный процесс **Vulnerability Management (VM)** с этапами:

1. **Инвентаризация** активов и компонентов.
2. **Выявление** уязвимостей (ручной поиск + автоматические сканеры SAST/DAST/SCA).
3. **Приоритизация** на основе CVSS, EPSS, наличия эксплойтов, значимости актива.
4. **Устранение** (патч, обновление, конфигурация, компенсирующая мера, принятие риска).
5. **Контроль** — повторное сканирование, проверка SLA, метрики.

Без VM-процесса уязвимости теряются: типичные проблемы — неполная классификация активов (в среднем классифицируется лишь ~37 % активов пилотной зоны), неактуальная инвентаризация (в 75 % компаний — устаревшая), нарушение SLA устранения и игнорирование контекста актива при приоритизации.

### 3.2. Классы компонентов по NIST SP 800-218 (SSDF)

При инвентаризации удобно опираться на классификацию NIST SSDF:

| Класс компонентов | Что туда входит | Где искать |
|---|---|---|
| Среды (контуры) разработки и поставки | dev / stage / prod, namespace’ы, кластеры | README, `.gitlab-ci.yml`, `values.yaml`, `docker-compose.yml` |
| Компоненты внутри среды и связи между ними | порты, маршруты Ingress, связи `app → DB / cache / broker` | манифесты K8s, `nginx.conf`, `docker ps`, `kubectl get svc/ingress` |
| Рабочие станции и ресурсы разработчиков | версии JDK/Node/Python, менеджеры пакетов | `.tool-versions`, `.nvmrc`, `.python-version`, README |
| Цепочка инструментов сборки | раннеры, плагины CI, версии Maven/Gradle/Node | `.gitlab-ci.yml`, `Makefile`, `pom.xml`, `package.json` |
| Код и конфигурации | исходники, IaC, конфиги безопасности | `src/`, `config/`, `Dockerfile`, `helm/`, `terraform/` |
| **Сторонние программные компоненты** | библиотеки, фреймворки, middleware, сервисы | lock-файлы, `docker-compose.yml`, Helm chart |
| Сведения о происхождении (provenance) | digest образа, теги, контрольные суммы, SBOM | `docker image inspect`, `cosign verify`, SBOM-файлы |
| Релизные материалы | теги релизных образов, артефакты | реестр контейнеров, `Chart.yaml` |
| Данные | какие данные обрабатывает сервис, где хранятся | API-спецификация, схема БД, `.env.example` |
| Роли и ответственность | владелец, контакт для триажа | README, CONTRIBUTING, CODEOWNERS |

В рамках этой лабораторной фокус — на классе **«Сторонние программные компоненты»** и частично на **«Сведения о происхождении»** и **«Компоненты внутри среды»**, потому что именно по ним выполняется ручной поиск CVE.

### 3.4. CVE, CVSS, CWE, CPE — что это и чем отличаются

- **CVE** — *идентификатор* конкретной уязвимости (например, `CVE-2024-12345`).
- **CVSS** — *оценка опасности* по шкале 0–10. Сейчас актуальны версии **3.1** и **4.0**. Векторная строка типа `CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H` описывает условия эксплуатации и воздействие. CVSS 4.0 ввёл различение влияния на «уязвимую систему» (VC/VI/VA) и «последующую систему» (SC/SI/SA), а также новую метрику Attack Requirements (AT).
- **CWE** — *классификация типа слабости* (например, `CWE-89: SQL Injection`). Один CWE может покрывать множество CVE.
- **CPE** — *стандартизованное имя продукта/версии* для машинного сопоставления, формат `cpe:2.3:a:vendor:product:version:...`. NVD ищет по CPE.

> **Важно:** CVSS — это про *максимальный потенциал* уязвимости, а не про реальную опасность для вашего стенда. Для приоритизации добавляйте контекст: **EPSS** (вероятность эксплуатации), **CISA KEV** (факт эксплуатации), **значимость актива**, **экспозиция** (доступность из интернета), **наличие компенсирующих мер**.

### 3.5. Алгоритм ручного поиска уязвимости

Для каждого компонента из инвентаризации:

1. Зафиксировать: имя, версию, источник версии (lock-файл / `docker image inspect` / `--version`).
2. Определить тип: библиотека / продукт от вендора / ОС-пакет или базовый образ.
3. Поиск:
   - **Библиотека** → экосистемная база (GHSA, OSV, PyPA, Go vuln DB, RustSec) → NVD.
   - **Продукт от вендора** → вендорские бюллетени (MSRC, Apache, nginx и т. д.) → NVD.
   - **ОС-пакет / базовый образ** → бюллетени дистрибутива (USN, Debian Security Tracker, RHSA) → OSV → NVD.
4. Зафиксировать CVE/CWE и идентификатор в БДУ ФСТЭК (если есть).
5. Опционально проверить EPSS и CISA KEV.
6. **Проверить применимость** к стенду: попадает ли ваша версия в диапазон уязвимых, включена ли уязвимая функциональность, доступна ли точка входа.
7. Оценить критичность: базовый CVSS + контекст актива + экспозиция.
8. Внести в реестр.

### 4.6. Системы управления уязвимостями. DefectDojo

**DefectDojo** — open-source платформа для централизованного управления находками безопасности. Поддерживает иерархию `Product → Engagement → Test → Finding`, импорт из десятков сканеров (Trivy, Bandit, Semgrep, ZAP, Burp, Nuclei, Snyk и др.), а также ручной ввод через UI или Generic Findings Import (JSON/CSV). В корпоративных пайплайнах DefectDojo обычно интегрируется с CI: каждый pipeline-job выгружает отчёт сканера в DefectDojo через API, после чего платформа дедуплицирует находки, ведёт жизненный цикл (Active / Verified / Mitigated / False Positive / Risk Accepted) и предоставляет метрики.

Альтернативы: **Faraday**, **OWASP Vulnerability Management Center**, **SecObserve**, **ArcherySec**.

## 5. Исходные условия и ограничения

- Используется выбранное приложение на `gitlab.digital.mephi.ru` (свой репозиторий курса).
- Анализ проводится **по фактически используемым компонентам** — то, что реально поднимается в стенде или объявлено в зависимостях проекта.
- Реестр уязвимостей должен содержать **10–15 записей** (репрезентативная выборка, а не «всё, что нашлось»):
  - минимум **2–3 относительно свежие** уязвимости (опубликованные в текущем или прошлом году);
  - минимум **3–5 уязвимостей** уровня **High / Critical** (если по выбранным компонентам такие в принципе существуют);
  - желательно покрыть **разные классы компонентов** (библиотека приложения, веб-сервер, БД/кэш, базовый образ).
- Для каждой записи реестра обязательно указывается **применимость к стенду** (версия / конфигурация / точка входа / экспозиция) — без этого находка считается невалидной.

- **Сценарий эксплуатации** — не «возможна атака», а конкретно: *«удалённый атакующий при доступе к эндпоинту X и при включённой функции Y может выполнить произвольный код в контексте процесса nginx-worker»*.
- **Рекомендация** — конкретная, проверяемая: *«обновить до версии ≥ 1.23.2»*, *«отключить модуль ngx_http_mp4_module в `nginx.conf`»*, *«ограничить доступ сетевым ACL к порту 5432»*, *«временно отключить эндпоинт `/api/v1/import`»*.
- **Применимость** — обязательно содержит обоснование: *почему именно эта уязвимость релевантна вашему стенду* (версия попадает в диапазон, функциональность включена, точка входа экспонирована).

В разделе «Выводы» отчёта:

- Топ-3 наиболее критичных уязвимостей для вашего стенда и **почему** именно эти (с учётом контекста, а не только CVSS).
- Какие компоненты «чистые», какие — «грязные».
- Приоритеты устранения (что чинить срочно, что — в плановом порядке, что — компенсировать).

### 6.2. Дополнительное задание (1 балл) — Загрузка результатов в DefectDojo

Внесите реестр в DefectDojo одним из способов.

**Вариант A. Ручной ввод через UI**

1. В созданном `Engagement` → `Tests` → `Add Test` → выбрать тип `Generic Findings Import` (или `Manual Entry`).
2. Внутри теста: `Findings` → `Add Finding`.
3. Для каждой записи реестра заполнить:
   - **Title** — короткое описание (например, `CVE-2022-22817: Pillow ImageMath RCE`);
   - **Severity** — Critical / High / Medium / Low / Info;
   - **CVE / Vulnerability ID** — идентификатор;
   - **CWE** — если знаете тип слабости;
   - **CVSSv3** — векторная строка;
   - **Description** — что за уязвимость, где находится в стенде;
   - **Mitigation** — рекомендация по устранению;
   - **Impact** — последствия для бизнеса/системы;
   - **References** — ссылки на NVD / GHSA / БДУ ФСТЭК;
   - **Active / Verified** — отметить статус.

**Вариант B. Импорт через Generic Findings Import (JSON)**

DefectDojo поддерживает формат `Generic Findings Import`. Подготовьте JSON по схеме:

```json
{
  "findings": [
    {
      "title": "CVE-2022-22817: Pillow ImageMath.eval RCE",
      "date": "2025-11-20",
      "cve": "CVE-2022-22817",
      "cwe": 94,
      "cvssv3": "CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H",
      "severity": "Critical",
      "description": "В Pillow < 9.0.0 функция ImageMath.eval() позволяет выполнить произвольный код Python через специально сформированное выражение. На стенде используется в обработчике /upload/avatar, доступном неавторизованным пользователям.",
      "mitigation": "Обновить Pillow до версии >= 9.0.0. Временная мера: ограничить /upload/avatar только для авторизованных пользователей и валидировать MIME-тип на уровне nginx.",
      "impact": "Полная компрометация контейнера приложения, доступ к БД через переменные окружения.",
      "references": "https://nvd.nist.gov/vuln/detail/CVE-2022-22817\nhttps://bdu.fstec.ru/vul/2022-XXXXX",
      "active": true,
      "verified": true,
      "false_p": false,
      "component_name": "pillow",
      "component_version": "8.3.0"
    }
  ]
}
```

Загрузить через UI: `Engagement → Import Scan Results → Scan Type: Generic Findings Import`.

Либо через API:

```bash
curl -X POST "http://localhost:8080/api/v2/import-scan/" \
  -H "Authorization: Token <YOUR_API_TOKEN>" \
  -F "scan_type=Generic Findings Import" \
  -F "engagement=<ENGAGEMENT_ID>" \
  -F "file=@findings.json" \
  -F "active=true" \
  -F "verified=true"
```

**Подтверждение выполнения:** скриншот раздела `Findings` в DefectDojo, где видно загруженные записи с корректной разбивкой по severity.

> 💡 *На что обратить внимание:* DefectDojo дедуплицирует находки по `title + severity + component`. Если повторно импортируете — старые findings перейдут в `Mitigated` или останутся `Active` в зависимости от настроек `Reimport`. На лабе достаточно одного импорта.

## 7. Оформление отчёта (по ГОСТ)

Оформить по [ГОСТ 7.32-2017](https://files.stroyinf.ru/Data/655/65555.pdf).

### Структура отчёта

1. **Титульный лист.**
2. **Оглавление.**
3. **Развертывание DefectDojo:**
   - требования (ПО / версии), команды запуска;
   - типовые ошибки и их исправления;
   - подтверждение работоспособности (скриншот или описание).
4. **Сбор компонентов и методика ручного поиска:**
   - какие источники использовались (compose / lock-файлы / `docker ps` и т. п.);
   - как определялись версии, как искались CVE;
   - **Таблица 1 — Компоненты для анализа.**
5. **Результаты: реестр уязвимостей:**
   - **Таблица 2 — Реестр уязвимостей компонентов** (10–15 записей);
   - краткие выводы: топ-3 критичных уязвимостей для вашего стенда и почему.
6. **Внесение результатов в DefectDojo** (если выполнялось дополнительное задание):
   - что сделали, что получилось, скриншоты Product / Engagement / Test / Findings.
7. **Список источников** (по ГОСТ).
