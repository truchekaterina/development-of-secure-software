## 3. Теоретическая часть

### 3.1. Место SAST в SSDLC

Безопасный жизненный цикл разработки (SSDLC) предусматривает встраивание контролей безопасности на каждой стадии. SAST относится к классу **white-box** проверок: ему доступен исходный код, конфигурации и схема сборки. На этапах SSDLC SAST встречается дважды:

- **Local SAST / IDE / pre-commit** — быстрая проверка на стороне разработчика, перед коммитом.
- **Централизованный SAST в CI** — полное сканирование при push/MR/build, с сохранением артефактов и загрузкой в систему управления уязвимостями.

Именно совмещение этих двух точек даёт **shift-left**: уязвимость ловится максимально близко к моменту его внесения, цена исправления минимальна.


### 3.2. Подходы статического анализа

Современный SAST редко работает на одном подходе — обычно комбинирует:

| Подход | Суть | Глубина | Производительность |
|---|---|---|---|
| **Pattern-based (сигнатурный)** | Регулярки/AST-паттерны | Низкая | Высокая |
| **AST analysis** | Анализ дерева разбора | Средняя | Высокая |
| **Control Flow Graph (CFG)** | Граф потока управления | Средняя | Средняя |
| **Data Flow Analysis (DFG)** | Поток данных между переменными | Высокая | Средняя |
| **Taint analysis** | Помечает «грязный» ввод и трассирует до sink | Высокая | Средняя |
| **Path-sensitive analysis** | Различает условия путей выполнения | Очень высокая | Низкая |
| **Symbolic execution** | Символическое выполнение путей | Очень высокая | Низкая |
| **Interprocedural / межпроцедурный** | Анализ через границы функций/модулей | Очень высокая | Низкая |

**Semgrep** в стандартном режиме — это pattern-based + AST анализ (быстрый, но не отслеживающий поток). В режиме `taint mode` Semgrep дополнительно трассирует помеченные данные source -> sink, что приближает его к классическим коммерческим SAST.

### 3.3. Source-to-Sink анализ — ключевая концепция

Идея в том, что уязвимость возникает не «в одной строке кода», а на **полном пути данных** от источника (source) до опасной точки использования (sink).

- **Source** — недоверенный ввод: параметры HTTP-запроса, заголовки, cookies, тело запроса, содержимое файлов, ответы внешних API, сообщения из очередей.
- **Sink** — операция, где данные могут навредить: SQL/NoSQL запрос, рендеринг HTML, `exec`/`system`, запись в файл по пути, `open()` с путём, redirect, `jsonpickle.loads`, и т. д.
- **Sanitizer / Validator** — функция, которая «очищает» данные: параметризованный запрос, контекстное экранирование, валидатор схемы.

Задача SAST — построить путь от source к sink и проверить, что на этом пути есть корректная функция sanitizer. Если sanitizer отсутствует или применён не в том контексте — это потенциальная уязвимость.

Именно поэтому при ручном триаже результатов вы должны для каждого срабатывания ответить на четыре вопроса:

1. Где **source** (точка попадания пользовательского ввода)?
2. Где **sink** (опасная операция)?
3. Есть ли между ними **sanitizer**? Достаточен ли он для контекста sink?
4. Достижим ли этот путь при реальных условиях работы приложения?

### 3.4. False Positive и False Negative — главная боль SAST

- **False Positive (FP)** — анализатор подсветил «уязвимость», которой по факту нет
- **False Negative (FN)** — реальная уязвимость, которую анализатор пропустил
- **True Positive (TP)** — подтверждённая ручным триажем уязвимость.

**Метрики качества SAST в CI** (отслеживайте их по проекту):

- **Precision** = TP / (TP + FP) — какая доля «алертов» оказалась настоящими;
- **Recall / Sensitivity** = TP / (TP + FN) — какую долю реальных уязвимостей мы поймали;
- **Triage time per finding** — сколько в среднем требуется на разбор одного срабатывания;
- **Доля suppressed (нолики)** — какую часть правил пришлось заглушить.

Чем выше доля FP — тем быстрее команда перестаёт читать отчёты SAST. Это главный риск некорректного внедрения.

### 3.5. Semgrep — устройство и синтаксис правил

**Semgrep** — open-source статический анализатор, написанный на OCaml.

**Минимальный пример правила:**

```yaml
rules:
  - id: python-sqlalchemy-text-string-fstring
    languages: [python]
    severity: ERROR
    message: >
      Использование f-string внутри sqlalchemy.text() приводит к SQL Injection.
      Параметризуйте запрос через :param-плейсхолдеры и .bindparams().
    metadata:
      cwe: "CWE-89: SQL Injection"
      owasp: "A03:2021 Injection"
      category: security
    pattern: |
      sqlalchemy.text(f"...")
```

**Пример taint-правила (source -> sink):**

```yaml
rules:
  - id: fastapi-sqlalchemy-taint-sql-injection
    languages: [python]
    severity: ERROR
    message: "Пользовательский ввод попадает в SQL-запрос без параметризации"
    mode: taint
    pattern-sources:
      - patterns:
          - pattern-either:
              - pattern: |
                  def $FUNC(..., $PARAM: str, ...):
                    ...
              - pattern: request.$ANYTHING
    pattern-sinks:
      - patterns:
          - pattern-either:
              - pattern: db.execute(text(f"...{$X}..."))
              - pattern: db.execute(text("..." + $X + "..."))
    pattern-sanitizers:
      - pattern: bindparam(...)
    metadata:
      cwe: "CWE-89"
      owasp: "A03:2021"
```

### 3.6. Стратегия подбора правил Semgrep под проект

Правила удобно подбирать по **трём направлениям покрытия**:

1. **По языку программирования** — общие правила для используемого языка: типовые небезопасные конструкции и опасные функции (`eval`, `exec`, `pickle.loads`, вызов `subprocess` с параметром `shell=True`).

2. **По классам уязвимостей (OWASP / CWE)** — правила, привязанные к перечням OWASP Top 10 и CWE Top 25: внедрение кода и запросов (Injection), межсайтовое выполнение сценариев (XSS), подделка запросов на стороне сервера (SSRF), небезопасные прямые ссылки на объекты (IDOR), криптографические ошибки, небезопасная десериализация.

3. **По специфике технологического стека** — правила под конкретный фреймворк или компонент: веб-фреймворки (`flask`, `django`, `fastapi`, `express`, `spring`, `react`, `nextjs`), системы объектно-реляционного отображения — ORM (`sqlalchemy`, `sequelize`, `hibernate`), очереди сообщений (`celery`, `kafka`), механизмы работы с токенами и авторизацией (JWT, OAuth), облачные SDK и т. д.

Итоговый набор правил для проекта формируется как объединение всех трёх направлений: это обеспечивает и базовую проверку языка, и покрытие известных классов уязвимостей, и учёт особенностей конкретного стека.

### 3.7. OWASP Top 10 (2025) — что из этого закрывает SAST

| Категория OWASP Top 10 (2025) | Покрытие SAST | Комментарий |
|---|---|---|
| A01 Broken Access Control | Слабое | Корректность контроля доступа определяется бизнес-правилами, а не текстом кода. Анализатор может заметить разве что отсутствие проверки роли там, где она ожидается по шаблону, но не способен понять, какому пользователю какой объект должен быть доступен. |
| A02 Security Misconfiguration | Частичное | Анализатор находит небезопасные настройки, если они заданы прямо в коде или конфигурационных файлах: включённый режим отладки, разрешение запросов с любых источников, открытые служебные страницы документации. |
| A03 Software Supply Chain | Не покрывается | Уязвимости в сторонних библиотеках и зависимостях — задача анализа состава ПО (SCA), а не статического анализа кода. |
| A04 Cryptographic Failures  | Хорошее | Анализатор уверенно выявляет применение устаревших алгоритмов хеширования, небезопасные режимы шифрования, секретные ключи, записанные прямо в коде, и недостаточную длину ключей. |
| A05 Injection | Отличное | Основная и наиболее сильная категория для статического анализа: выявляются места, где недоверенные данные попадают в SQL- и NoSQL-запросы, системные команды, запросы к каталогам LDAP и выражения XPath. |
| A06 Insecure Design  | Не покрывается | Это дефекты на уровне архитектуры и проектных решений. Их выявление требует моделирования угроз, а не анализа исходного кода. |
| A07 Authentication Failures | Слабое | Анализатор может заметить отдельные технические ошибки — например, разбор токена без проверки подписи или секреты в коде, — но не способен оценить корректность самой логики аутентификации и управления сессиями. |
| A08 Software or Data Integrity Failures  | Частичное | Опасные операции вроде небезопасной десериализации или динамического выполнения кода выявляются. Проверку же целостности данных и протоколов обмена статический анализ оценить не может. |
| A09 Security Logging & Alerting Failures  | Не покрывается | Достаточность журналирования и мониторинга проверяется в рамках аудита соответствия требованиям и ручного анализа кода, а не автоматическим сканированием. |
| A10 Mishandling of Exceptional Conditions  | Частичное | Анализатор может выявить отдельные опасные шаблоны — например, когда при сбое защитный механизм по ошибке разрешает операцию вместо того, чтобы её заблокировать. |

### 3.8. Интеграция SAST в CI/CD

Хорошая SAST-интеграция в pipeline соответствует следующим принципам:
Вот переписанный фрагмент — нормальным русским языком, без жаргона:

- **Неблокирующий первый запуск.** На старте проверка не должна прерывать сборку при обнаружении уязвимостей (в GitLab CI это задаётся параметром `allow_failure: true`). Это даёт команде время разобрать уже накопившиеся замечания, не останавливая текущую работу.

- **Сравнение с базовой версией.** После первого полного запуска прерывать сборку имеет смысл только из-за уязвимостей в **новом или изменённом коде**, а не во всей кодовой базе. Semgrep поддерживает такой режим через указание базовой ветки для сравнения (`semgrep ci --baseline-ref`). Это особенно важно для больших проектов с большим объёмом унаследованного кода — иначе разработчиков «затопит» замечаниями к коду, который они не трогали.

- **Быстрое выполнение проверки.** Этап SAST должен укладываться в общее допустимое время сборки — обычно это не более 5–10 минут для среднего проекта. При необходимости проверку запускают параллельно с другими этапами.

- **Сохранение отчётов в машиночитаемом формате.** Результаты сохраняются как артефакты в форматах SARIF или JSON. Это единый формат, который понимают система управления уязвимостями (DefectDojo), панель безопасности GitLab и плагины для среды разработки.

- **Файл исключений `.semgrepignore`.** В него выносятся каталоги, которые не нужно проверять: тесты, миграции базы данных, сторонние зависимости, сжатые (минифицированные) файлы JavaScript.

- **Контрольный этап качества (Quality Gate).** Отдельный шаг конвейера, который прерывает сборку только в определённых случаях — например, при появлении новых уязвимостей высокого или критического уровня либо при превышении заданного порога по их количеству.

---

Типичный пример pre-build SAST job в GitLab CI:

```yaml
semgrep_scan:
  stage: pre-build
  image: returntocorp/semgrep:latest
  variables:
    SEMGREP_RULES: "p/owasp-top-ten p/cwe-top-25 p/python"
  script:
    - mkdir -p semgrep/reports
    - semgrep scan --config semgrep/semgrep.yml --json --output semgrep/reports/report.json .
  artifacts:
    paths: [semgrep/reports/]
    expire_in: 1 week
  allow_failure: true
```

## 4. Исходные условия и ограничения

- Используется выбранное приложение из репозитория `http://10.0.0.10` (тот же проект, что и в лабораторных работах №1, 3, 4).

- Анализ выполняется **на основном языке** приложения. Если язык один — работаем только с ним. Если языков несколько (например, серверная часть на Python и клиентская на JavaScript) — необходимо проанализировать оба.

- Правила Semgrep подбираются **самостоятельно** и сохраняются **локально в репозитории проекта** (в каталоге `semgrep/rules/`). Использование облачного сервиса Semgrep Cloud (запуск `semgrep ci` с авторизацией в облаке) **не допускается** — сборка должна быть воспроизводимой и работать без доступа к интернету.

- Минимальный набор правил: **5 правил по языку, 5 правил по классам уязвимостей (OWASP / CWE) и 5 правил под технологический стек** — всего не менее 15.

- Для каждого подтверждённого срабатывания (минимум 3–5 штук) обязательно описываются: **в чём состоит риск, как уязвимость может быть использована и как её следует устранить**.

- Сравнение результатов «до» и «после» настройки обязательно: подсчитывается количество найденных проблем по уровням опасности, а также доля ложных срабатываний до и после настройки правил.

#### Часть 2. Подбор правил Semgrep

Создайте структуру в репозитории:

```
semgrep/
├── rules/
│   ├── language/        # правила под Python / JS / ...
│   ├── owasp/           # OWASP Top 10 / CWE Top 25
│   ├── stack/           # FastAPI / SQLAlchemy / JWT / Crypto
│   └── custom/          # ваши собственные правила (минимум 1)
├── semgrep.yml          # головной конфиг
└── reports/
    └── .gitkeep
```

**Источники готовых правил:**

- Реестр Semgrep: <https://semgrep.dev/explore>
- Репозиторий правил: <https://github.com/semgrep/semgrep-rules>
- Документация синтаксиса: <https://semgrep.dev/docs/writing-rules/overview>

**Минимальный набор (≥ 15 правил):**

| Категория правил | Минимум | Примеры (для Python / FastAPI) |
|---|---|---|
| Правила по языку | 5 | `dangerous-system-call`, `eval-detected`, `exec-detected`, `assert-detected`, `weak-cryptographic-hash` |
| Правила по классам уязвимостей (OWASP / CWE) | 5 | `sql-injection-text`, `command-injection-os-system`, `pickle-unsafe`, `unverified-jwt-decode`, `hardcoded-secret` |
| Правила под технологический стек | 5 | `fastapi-cors-allow-all`, `sqlalchemy-text-fstring`, `jwt-none-algorithm`, `flask-debug-true`, `requests-no-verify` |
| **Собственные правила** | ≥ 1 | Правило, написанное самостоятельно под особенности конкретного проекта |

> **Скачивание правил вручную** (из-за ограничений сети Registry может быть недоступен):

**Головной конфиг `semgrep/semgrep.yml`:**

```yaml
rules:
  - include: rules/language/*.yaml
  - include: rules/owasp/*.yaml
  - include: rules/stack/*.yaml
  - include: rules/custom/*.yaml
```

**Custom-правило (обязательное)** — напишите минимум одно своё правило, специфичное для бизнес-логики или конвенций вашего проекта. Пример:

```yaml
# semgrep/rules/custom/no-print-in-prod-handlers.yaml
rules:
  - id: no-print-in-fastapi-handlers
    languages: [python]
    severity: WARNING
    message: >
      В FastAPI-обработчиках вместо print() используйте logger
      (print попадает в stdout без структуры, не маскирует PII и ломает аудит).
    pattern-either:
      - pattern: |
          @router.$DECORATOR(...)
          def $FUNC(...):
              ...
              print(...)
      - pattern: |
          @app.$DECORATOR(...)
          def $FUNC(...):
              ...
              print(...)
    metadata:
      category: best-practice
      cwe: "CWE-532: Insertion of Sensitive Information into Log File"
      owasp: "A09:2025 Security Logging Failures"
```

#### Часть 4. Анализ и проверка результатов

Для каждой найденной проблемы определите её статус:

- **Подтверждённое срабатывание (True Positive, TP)** — реальная уязвимость: путь от точки попадания данных (источника) до опасной операции (приёмника) не прерывается проверкой или очисткой.
- **Ложное срабатывание (False Positive, FP)** — анализатор отметил проблему, но по факту её нет: данные очищаются, путь недостижим при работе приложения, либо это тестовый код.
- **Требует уточнения** — для вывода нужна консультация с владельцем кода или архитектором.

Из подтверждённых срабатываний выберите **минимум 3–5** наиболее значимых и оформите по ним подробный разбор (Таблица 2).

### 5.2. Дополнительное задание (+1 балл) — Интеграция Semgrep в GitLab CI и отправка результатов в DefectDojo

В рамках дополнительного задания необходимо встроить статический анализ в конвейер сборки проекта и настроить автоматическую передачу его результатов в систему управления уязвимостями DefectDojo. Конкретные команды, синтаксис файла `.gitlab-ci.yml` и параметры запросов студенты подбирают самостоятельно, опираясь на официальную документацию GitLab CI/CD, Semgrep и DefectDojo.

#### Часть 1. Запуск Semgrep в GitLab CI

Необходимо добавить в конвейер проекта задачу (job), которая:

- запускается на этапе до сборки (`pre-build`);
- использует подготовленную локальную конфигурацию Semgrep (`semgrep/semgrep.yml`);
- выполняет анализ с зафиксированной версией Semgrep (не `latest`), чтобы запуски были воспроизводимыми;
- формирует отчёт в форматах JSON и SARIF;
- сохраняет отчёты как артефакты GitLab со сроком хранения 1 неделя;
- работает в неблокирующем режиме — наличие найденных проблем не должно приводить к сбою конвейера;
- не отправляет телеметрию во внешние сервисы.

После запуска конвейера необходимо убедиться, что задача выполняется, отчёт формируется, артефакты сохраняются, а конвейер не завершается ошибкой из-за найденных проблем.

#### Часть 2. Контрольный этап качества (Quality Gate)

Необходимо добавить отдельную задачу, которая прерывает конвейер при определённых условиях — например, при обнаружении проблем критического уровня. Логику и пороги срабатывания студент определяет самостоятельно и обосновывает в отчёте: что именно блокирует сборку и почему выбран такой критерий.

#### Часть 3. Передача результатов в DefectDojo

Необходимо настроить автоматическую загрузку отчёта Semgrep в DefectDojo. Для этого требуется:

- в DefectDojo подготовить структуру для приёма результатов (Product, Engagement) и получить ключ доступа к API;
- в настройках GitLab CI/CD создать переменные для адреса DefectDojo, ключа API и идентификатора Engagement; ключ API должен быть защищён (masked + protected);
- добавить в конвейер задачу, которая запускается после анализа Semgrep, использует сформированный отчёт и передаёт его в DefectDojo через API;
- обеспечить, чтобы при повторных запусках конвейера результаты обновлялись, а не дублировались (обратите внимание на разницу между методами импорта и повторного импорта в API DefectDojo).

После выполнения необходимо убедиться, что результаты анализа появляются в DefectDojo с корректными уровнями опасности и привязкой к нужному Engagement, а повторный запуск конвейера не создаёт дубликаты.

#### Что приложить в отчёт

- содержимое `.gitlab-ci.yml` (задачи запуска Semgrep, контрольного этапа и загрузки в DefectDojo);
- скриншот успешно выполненного конвейера;
- скриншот задачи запуска Semgrep с выводом версии и сводки по уровням опасности;
- список сохранённых артефактов и фрагмент сформированного отчёта;
- скриншот списка переменных в настройках GitLab CI/CD (значения замаскированы);
- скриншот ответа DefectDojo в логе задачи загрузки;
- скриншот раздела с загруженными результатами в DefectDojo (Engagement -> Tests -> Findings).

## 7. Может быть интересно

- **«Найдено более 500 проблем, непонятно, что с этим делать».** Для первого запуска на унаследованном коде это нормально. Правильный порядок действий: (1) отключите правила, по которым возникает больше 10 однотипных ложных срабатываний; (2) добавьте в файл исключений `.semgrepignore` каталоги с тестами, миграциями и сторонними зависимостями; (3) сосредоточьтесь на проблемах критического уровня (`severity: ERROR`). Также можно использовать режим сравнения с базовой версией (`semgrep ci --baseline-ref`), чтобы видеть только новые проблемы.

- **«Semgrep не нашёл уязвимость, которая точно есть».** Это типичный пропуск (ложноотрицательное срабатывание). Возможные причины: нестандартная опасная операция — например, SQL-запрос собирается по частям в нескольких функциях; отсутствует правило под используемый фреймворк; требуется перейти в режим анализа потока данных (`taint mode`). Это **не ошибка Semgrep**, а ограничение анализа по шаблонам.

- **«Срабатывание подавлено комментарием `# nosemgrep` без пояснения».** Это сокрытие проблемы, а не её решение. Любое подавление должно сопровождаться комментарием с указанием причины и, по возможности, ссылкой на задачу в трекере.

- **«Запуск выполнен из каталога, где лежит `.venv`».** Semgrep начнёт проверять сторонние пакеты, и время анализа вырастет в десятки раз. Каталог виртуального окружения обязательно нужно внести в `.semgrepignore`.

- **«В DefectDojo при каждом запуске создаётся новый тест».** Используйте метод `reimport-scan/`, а не `import-scan/` — он обновляет существующий тест, а не создаёт новый.

## Приложение А. Шаблон custom-правила Semgrep

```yaml
# semgrep/rules/custom/fastapi-route-without-auth.yaml
rules:
  - id: fastapi-route-without-auth-dependency
    languages: [python]
    severity: WARNING
    message: >
      FastAPI-обработчик не использует Depends(get_current_user) и не помечен
      явным @public_endpoint. По умолчанию каждая ручка должна быть
      авторизованной — иначе риск Broken Access Control (A01:2025).
    patterns:
      - pattern-either:
          - pattern: |
              @$ROUTER.get(...)
              def $F(...):
                ...
          - pattern: |
              @$ROUTER.post(...)
              def $F(...):
                ...
          - pattern: |
              @$ROUTER.put(...)
              def $F(...):
                ...
          - pattern: |
              @$ROUTER.delete(...)
              def $F(...):
                ...
      - pattern-not: |
          @$ROUTER.$M(...)
          def $F(..., $U: User = Depends(get_current_user), ...):
            ...
      - pattern-not: |
          @public_endpoint
          @$ROUTER.$M(...)
          def $F(...):
            ...
    metadata:
      category: security
      cwe: "CWE-862: Missing Authorization"
      owasp: "A01:2025 Broken Access Control"
      confidence: MEDIUM
```

## Приложение В. Полный `.gitlab-ci.yml` (SAST + DefectDojo)

```yaml
stages:
  - pre-build
  - quality-gate

variables:
  SEMGREP_VERSION: "1.95.0"
  REPORT_DIR: "semgrep/reports"

# ---------- SAST ----------
semgrep_scan:
  stage: pre-build
  image: returntocorp/semgrep:${SEMGREP_VERSION}
  variables:
    SEMGREP_METRICS: "off"
  before_script:
    - mkdir -p ${REPORT_DIR}
    - semgrep --version
  script:
    - |
      semgrep scan \
        --config semgrep/semgrep.yml \
        --json   --output ${REPORT_DIR}/report.json \
        --sarif  --output ${REPORT_DIR}/report.sarif \
        --metrics=off --verbose .
    - |
      jq -r '[.results[] | .extra.severity] | group_by(.) |
             map({sev: .[0], count: length}) | .[] |
             "\(.sev): \(.count)"' ${REPORT_DIR}/report.json
  artifacts:
    when: always
    paths: [semgrep/reports/]
    reports:
      sast: ${REPORT_DIR}/report.sarif
    expire_in: 1 week
  allow_failure: true

# ---------- DefectDojo upload ----------
upload_to_defectdojo:
  stage: pre-build
  needs: [semgrep_scan]
  image: alpine:3.20
  before_script:
    - apk add --no-cache curl jq
  script:
    - |
      curl --silent --show-error --fail-with-body \
        --request POST "${DEFECTDOJO_URL}/api/v2/reimport-scan/" \
        --header "Authorization: Token ${DEFECTDOJO_API_KEY}" \
        --form "scan_type=Semgrep JSON Report" \
        --form "engagement=${DEFECTDOJO_ENGAGEMENT_ID}" \
        --form "file=@${REPORT_DIR}/report.json" \
        --form "active=true" \
        --form "verified=false" \
        --form "minimum_severity=Low" \
        --form "scan_date=$(date +%Y-%m-%d)" \
        --form "tags=semgrep,ci,${CI_COMMIT_REF_SLUG}" \
        --form "commit_hash=${CI_COMMIT_SHA}" \
        --form "branch_tag=${CI_COMMIT_REF_NAME}" | jq .
  rules:
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
  allow_failure: true

# ---------- Quality Gate ----------
semgrep_gate:
  stage: quality-gate
  needs: [semgrep_scan]
  image: alpine:3.20
  before_script:
    - apk add --no-cache jq
  script:
    - |
      CRITICAL=$(jq '[.results[] | select(.extra.severity == "ERROR")] | length' ${REPORT_DIR}/report.json)
      echo "Critical (ERROR) findings: $CRITICAL"
      if [ "$CRITICAL" -gt 0 ]; then
        echo " Pipeline blocked: $CRITICAL critical findings"
        exit 1
      fi
  rules:
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
  allow_failure: false
```
