## Теоретическая часть

### Базовые понятия информационной безопасности

Любая практическая задача в ИБ строится вокруг трёх вопросов: **что защищать**, **от чего защищать** и **как защищать**.
Лабораторная отвечает на первые два — нужно сначала понять, *что вообще есть в системе* и *где у этого слабые места*.

**Актив (asset)** — всё, что имеет ценность для организации и нуждается в защите. В контексте веб-приложения это:

- **Данные** — персональные данные пользователей, платёжные реквизиты, хеши паролей, сессионные токены, API-ключи.
- **Бизнес-логика** — алгоритмы скидок, рекомендаций, скоринга — то, что отличает приложение от конкурентов.
- **Инфраструктура** — серверы БД, кластер Kubernetes, CI/CD-пайплайн, образы контейнеров, доменное имя, TLS-сертификаты.
- **Репутация и доверие** — бренд, SLA, рейтинги.

> Без понимания, **что** ценно, защищать что-либо бессмысленно: ресурсы конечны, угрозы бесконечны.

**Уязвимость (vulnerability)** — свойство системы, которое позволяет реализовать угрозу. Уязвимость существует
**объективно**, независимо от того, знает о ней кто-то или нет. Важно: уязвимость ≠ баг.

- *Баг* — несоответствие поведения программы тому, что задумывал разработчик (опечатка в кнопке, кривая сумма в корзине).
- *Уязвимость* — слабость, через которую можно нарушить безопасность. Слабая парольная политика «4 символа» —
  это уязвимость, но не баг: код работает ровно так, как задумано, проблема в самом замысле.

**Угроза (threat)** — потенциально возможное событие, нарушающее свойства ИБ. Угроза — это «что *может* произойти».
Источники угроз бывают **антропогенные** (хакеры, инсайдеры, ошибки сотрудников),
**техногенные** (сбои, отказы) и **природные** (пожар, затопление).

**Атака (attack)** — конкретное действие, в котором кто-то применил уязвимость для реализации угрозы.
**Отсутствие атаки не означает безопасность** — оно означает только, что пока не сошлись все условия.

**Риск (risk)** — комбинация **вероятности** реализации угрозы и **ущерба** от неё:

```
Риск ≈ Вероятность × Ущерб
```

С риском можно делать четыре вещи: **принять** (если ущерб мал), **снизить** (внедрить меры защиты),
**передать** (страхование, аутсорсинг) или **избежать** (отказаться от рискованного функционала).

### Свойства информационной безопасности

Защищённость информации измеряется через набор свойств. Базовая триада — **CIA**:

| Свойство | Что означает | Пример нарушения в веб-приложении |
|---|---|---|
| **Confidentiality** (конфиденциальность) | Информация доступна только тем, кому положено | Утечка БД с ПДн через SQLi |
| **Integrity** (целостность) | Информация неизменна или изменения обнаруживаются | Подмена цены товара через DOM-tampering |
| **Availability** (доступность) | Информация доступна, когда нужна | DDoS на API в Чёрную пятницу |

Расширенный набор включает **аутентичность** (подлинность субъекта), **подотчётность** (возможность проследить
действие до субъекта), **неотказуемость** (нельзя отказаться от совершённого действия) и **надёжность** (предсказуемое поведение).

В этой лабораторной вы будете **косвенно** оценивать, какие свойства каких активов потенциально нарушаются
на разных точках поверхности атаки.

### Поверхность атаки

**Поверхность атаки (attack surface)** — совокупность всех точек системы, через которые недоверенный субъект
может попытаться войти, передать данные или получить ответ.

> Поверхность атаки — это **не уязвимости**. Это **потенциальные точки входа**, в которых уязвимости *могут быть*.

Поверхность атаки традиционно делят на четыре уровня:

| Уровень          | Что входит                                                 | Примеры на веб-приложении                                                                       |
| ---------------- | ---------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| **Физический**   | Серверы, рабочие станции, носители, кабельные сети         | Кража ноутбука разработчика, физический доступ к серверной                                      |
| **Сетевой**      | Открытые порты, протоколы, адреса, DNS-записи, поддомены   | Открытый наружу `5432` PostgreSQL, забытый management-порт, поддомен `staging.app.com` без auth |
| **Программный**  | Точки взаимодействия с кодом, обработка ввода, зависимости | Поля форм, параметры URL, заголовки HTTP, cookie, загрузка файлов, библиотеки в `package.json`  |
| **Человеческий** | Сотрудники, подрядчики, инсайдеры                          | Фишинг разработчика → утечка GitLab-токена; подрядчик со слабым VPN-паролем                     |

**Главное правило:** чем больше поверхность — тем выше вероятность, что найдётся слабое место.
Защита всегда начинается с её **измерения и сокращения**.

В этой лабораторной вы детально работаете с **сетевым** и **программным** уровнями;
**человеческий** — справочно (учётки, секреты), **физический** — за рамками рассмотрения.

### Вектор атаки

**Вектор атаки (attack vector)** — конкретный путь и метод, которым атакующий доставляет полезную нагрузку
через поверхность атаки для реализации угрозы. Если поверхность — это «двери», то вектор — это
«как именно атакующий попробует пройти через конкретную дверь».

Вектор описывается четырьмя компонентами:
[[Реальный Вектор атаки]]

| Компонент | Что описывает | Пример |
|---|---|---|
| Точка входа | Какой элемент поверхности атаки используется | Эндпоинт `POST /api/login` |
| Канал доставки | Как именно попадает payload | HTTP POST с JSON в теле |
| Полезная нагрузка (payload) | Что отправляет атакующий | `{"email":"' OR 1=1-- ","password":"x"}` |
| Цель эксплуатации | Чего хочет добиться атакующий | Обход аутентификации → доступ к admin-панели |

Из этого вытекает важное определение:

> **Поверхность атаки = совокупность всех потенциально доступных векторов атаки.**

Одна точка поверхности обычно даёт **несколько разных векторов**. Например, форма поиска `/search?q=`:

- Reflected XSS — `?q=<script>...</script>`,
- SQL-инъекция — `?q=test' UNION SELECT ...`,
- ReDoS — патологический регэксп в `q`,
- утечка структуры БД через сообщения об ошибках.

В реальной атаке векторы **складываются в цепочку (kill chain)**: разведка → точка входа → эскалация привилегий → доступ к секретам → эксфильтрация. 
Защитнику важно перекрывать **звенья цепочки**, а не отдельные векторы.

### Границы доверия

**Граница доверия (trust boundary)** — место, где данные или управление переходят от менее доверенного субъекта к более доверенному. На границе **всегда** должна стоять защитная мера: валидация, аутентификация, авторизация или санитизация.

Типичные границы доверия в веб-приложении:

| Граница | Где проходит | Что должно стоять на границе |
|---|---|---|
| Клиент <-> Сервер | Браузер пользователя <-> backend | Серверная валидация, аутентификация, security headers |
| Сервер <-> База данных | Backend <-> СУБД | Минимальные права у БД-пользователя, параметризованные запросы |
| Сервис <-> Сервис (внутри системы) | Микросервис A <-> Микросервис B | Взаимная аутентификация (mTLS), service-to-service токены |
| Приложение <-> Внешнее API | Backend <-> платёжный провайдер | Whitelist URL, проверка TLS-сертификата, ротация ключей |
| Контейнер <-> Хост | Docker-контейнер <-> ОС | Non-root user, drop capabilities, read-only FS |
| Разработчик <-> Репозиторий | Локальный git <-> GitLab | Pre-commit secrets scan, code review, branch protection |

> Типичная ошибка начинающих — считать, что граница доверия одна: «клиент — сервер».
> Внутри системы **всегда** есть свои границы, и компрометация одного компонента не должна автоматически
> давать доступ ко всем остальным (принцип эшелонированной защиты — *defense in depth*).

### 5.6 Принципы работы с поверхностью атаки

Поверхностью атаки управляют по трём принципам — **строго в этом порядке**:

| | INVENTORY | MINIMIZATION | HARDENING |
|---|---|---|---|
| Что | Знаем всё, что у нас есть | Убираем всё, что не нужно | Усиливаем то, что должно остаться |
| Как | Сканирование, SBOM, ASM-инструменты | Удаление legacy, закрытие портов, чистка зависимостей | MFA, WAF, патчи, secure defaults |

**1. Inventory (учёт).** Нельзя защитить то, о чём не знаешь. На практике — активные сканы периметра
(`nmap`, `masscan`), пассивный сбор (DNS-логи, NetFlow), генерация SBOM (CycloneDX/SPDX),
внешние ASM-инструменты (Censys, Defender EASM, Kaspersky Digital Footprint).

**2. Minimization (сокращение).** Что не существует — то нельзя атаковать. Закрытие неиспользуемых портов,
удаление legacy-эндпоинтов, выключение debug-интерфейсов на проде, удаление тестовых пользователей,
сокращение зависимостей. Опирается на принципы **Least Privilege** (минимум прав)
и **Need-to-Know** (минимум осведомлённости).

**3. Hardening (укрепление).** То, что должно остаться, защищаем качественно: secure defaults у фреймворков,
MFA на критичных доступах, регулярный patch management, WAF/RASP/EDR, мониторинг.

В этой лабораторной вы выполняете **только Inventory** — первый и самый недооценённый шаг.

## Методика анализа поверхности атаки

### Источники информации: где искать

Поверхность атаки восстанавливается из четырёх параллельных источников. **Ни одного нельзя пропустить** —
каждый показывает систему под своим углом, и часто они расходятся между собой.

**1. Документация проекта** — что *задумывал* автор:

- `README.md`, `INSTALL.md`, `docs/`, wiki репозитория
- описание архитектуры, переменные окружения, примеры запуска

**2. Декларативные конфиги** — что *написано* в инфраструктуре:

| Файл | Что показывает |
|---|---|
| `docker-compose.yml`, `Dockerfile` | сервисы, порты, тома, переменные, базовые образы |
| `.env`, `.env.example` | какие секреты нужны приложению |
| `nginx.conf`, `apache.conf`, `Caddyfile` | маршруты, заголовки, проксирование |
| `requirements.txt`, `package.json`, `go.mod`, `pom.xml` | прямые зависимости |
| `*-lock.json`, `poetry.lock`, `Gemfile.lock` | транзитивные зависимости |
| `application.yml`, `settings.py`, `config/database.yml` | настройки приложения |
| `migrations/`, `schema.sql` | структура БД, имена таблиц и колонок |
| `.gitlab-ci.yml`, `.github/workflows/` | CI/CD, секреты пайплайна |

**3. Код приложения** — что *реально делает* система:

- `urls.py` / `routes.js` / `@RestController` — маршруты и эндпоинты
- middleware и декораторы — где стоят проверки auth/permissions
- модели — что хранится в БД
- точки парсинга ввода (`request.GET`, `req.body`, `BindJSON`) — кандидаты на инъекции

**4. Развернутый стенд** — что *происходит на самом деле*:

- `docker compose ps` — какие контейнеры реально запущены
- `ss -tlnp` — какие порты реально слушаются на хосте
- DevTools браузера: вкладки `Network` и `Application → Cookies/Storage`
- `curl -i` к эндпоинтам — какие заголовки, какие коды ответов
- логи: `docker compose logs <service>`

### Алгоритм работы

Рекомендуемый порядок:

```
1. Прочитать README, понять, что вообще делает приложение     
2. Открыть docker-compose.yml / Dockerfile                     
   ├─ Какие сервисы (компоненты системы)
   ├─ Какие порты и куда биндятся                              
   ├─ Какие тома и где данные
   └─ Какие переменные окружения                               
3. Поднять стенд и сверить с конфигом
   ├─ docker compose ps
   ├─ ss -tlnp
   └─ curl на основные эндпоинты
4. Открыть код, найти роуты                                    
   ├─ Какие пути есть
   ├─ Какие принимают данные
   └─ Где требуется авторизация
5. Найти секреты в .env, .env.example, конфигах                
6. Определить границы доверия и сформулировать риски           
```

### Практические команды

#### Сетевая поверхность

```bash
# Что запущено в Docker
docker compose ps
docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Ports}}\t{{.Status}}"

# Какие порты реально слушаются на хосте
ss -tlnp                                          # все TCP listening
sudo lsof -iTCP -sTCP:LISTEN -P                   # альтернатива
netstat -tlnp                                     # старый вариант

# Какие порты публикует контейнер
docker port <container_name>

# Внутренние сети Docker
docker network ls
docker network inspect <network_name>

# Что отвечает на порте
curl -i http://localhost:8080/
curl -I http://localhost:8080/                    # только заголовки
curl -i -X OPTIONS http://localhost:8080/api/     # CORS, разрешённые методы
```

> По заголовкам ответа (`Server`, `X-Powered-By`, `Set-Cookie`) часто можно понять, на каком фреймворке
> работает приложение. Это уже само по себе информация для оценки

#### Программная поверхность

```bash
# Django
grep -rn "path(\|re_path(\|url(" --include="*.py" .

# Flask / FastAPI
grep -rn "@app\.\|@router\." --include="*.py" .

# Express.js / Node.js
grep -rn "app\.\(get\|post\|put\|delete\|patch\)\|router\." --include="*.js" --include="*.ts" .

# Spring (Java)
grep -rn "@RequestMapping\|@GetMapping\|@PostMapping" --include="*.java" .

# Laravel / Symfony (PHP)
grep -rn "Route::\|@Route" --include="*.php" .

# Точки приёма пользовательского ввода
grep -rn "request\.GET\|request\.POST\|request\.body\|req\.body\|req\.query" .

# Загрузка файлов
grep -rn "multipart\|FileField\|MultipartFile\|FormData" .

# Прямые SQL-запросы (кандидаты на SQLi)
grep -rn "execute(\|raw(\|cursor\.execute\|query(" --include="*.py" --include="*.js" .
```

#### Зависимости

```bash
cat requirements.txt              # Python pip
cat package.json                  # Node.js (прямые)
cat package-lock.json | head      # Node.js (включая транзитивные)
cat go.mod                        # Go
cat composer.json                 # PHP
cat Gemfile.lock                  # Ruby

# Сколько всего зависимостей с учётом транзитивных
pip list 2>/dev/null | wc -l
npm ls --all 2>/dev/null | wc -l
```

#### Поиск секретов

```bash
# Файлы с потенциальными секретами
ls -la .env* config/secrets* secrets/* 2>/dev/null
find . -name "*.key" -o -name "*.pem" -o -name "*.p12" 2>/dev/null

# Поиск по ключевым словам в коде
grep -rEn "(password|passwd|secret|api[_-]?key|token|private[_-]?key)\s*[=:]" \
  --include="*.py" --include="*.js" --include="*.yml" --include="*.env*" .

# Hardcoded значения, похожие на секреты (строки 32+ символов из base64/hex алфавита)
grep -rEn "['\"]([A-Za-z0-9+/]{32,}|[a-f0-9]{32,})['\"]" --include="*.py" --include="*.js" .

# Проверка, что секреты не закоммичены
git log --all --full-history -- .env 2>/dev/null
```

> Системно секреты ищут инструментами: `gitleaks`, `trufflehog`, `detect-secrets`. Но на текцщий момент хватит `grep` или речной проверки.

#### Чтение схемы БД

```bash
# PostgreSQL
docker compose exec db psql -U <user> -d <db> -c "\dt"           # список таблиц
docker compose exec db psql -U <user> -d <db> -c "\d users"      # структура таблицы
docker compose exec db psql -U <user> -d <db> -c "\du"           # пользователи и их права

# MySQL / MariaDB
docker compose exec db mysql -u <user> -p<pass> -e "SHOW TABLES" <db>
docker compose exec db mysql -u <user> -p<pass> -e "DESCRIBE users" <db>
```

#### 5.1 Условия развёртывания и границы рассмотрения

Текстом: где запущено приложение, что считается «внешней» средой, а что «внутренней»,
какие части системы вы рассматриваете, что — нет.

> **Пример формулировки:** «Приложение развёрнуто локально через Docker Compose.
> Внешняя среда — взаимодействие пользователя через браузер и запросы с хоста на `localhost`.
> Внутренняя среда — Docker-сеть `appnet`, в которой работают сервисы БД и кэша.
> Физическая безопасность хоста и человеческий фактор за рамками рассмотрения.»

#### 5.2 Состав системы и используемые технологии

| Компонент | Программное решение | Назначение | Где запущено |
|---|---|---|---|
| Frontend | React 18 | UI пользователя | контейнер `web` |
| Backend | Django 4.2 (Python 3.11) | API и серверный рендеринг | контейнер `app` |
| База данных | PostgreSQL 15 | Хранение данных пользователей | контейнер `db` |
| Кэш | Redis 7 | Сессии, очередь задач | контейнер `cache` |
| Reverse proxy | nginx 1.25 | Терминация TLS, отдача статики | контейнер `nginx` |

#### 5.3 Сетевые точки доступа (порты и адреса)

| Адрес и порт | Компонент | Протокол | Назначение | Доступно извне |
|---|---|---|---|---|
| `0.0.0.0:8080` | nginx | HTTP | Веб-интерфейс | да |
| `127.0.0.1:5432` | PostgreSQL | TCP | Доступ к БД | нет (только хост) |
| `0.0.0.0:6379` | Redis | TCP | Биндинг наружу — наблюдение | да (избыточно) |

> Подсказка: фактическую картину портов берите из `docker compose ps` и `ss -tlnp` на хосте,
> сверяйте с `docker-compose.yml`. Расхождения — повод для отдельной строки в отчёте.

#### 5.4 Точки входа в приложение

| Категория | Путь / страница | Тип запроса | Авторизация | Принимаемые данные |
|---|---|---|---|---|
| Вход | `/login` | POST | нет | `email`, `password` |
| Поиск | `/search` | GET | нет | query-параметр `q` |
| Профиль | `/api/users/{id}` | GET | да (JWT) | path-параметр `id` |
| Загрузка файлов | `/upload` | POST | да | `multipart/form-data` |
| Админка | `/admin/` | GET/POST | да (admin) | формы, CSRF-токен |

#### 5.5 Доступ, роли, данные и секреты

**5.5.1 Способы аутентификации и роли**

| Способ входа | Где используется | Что передаётся |
|---|---|---|
| Сессионная cookie | браузер <-> backend | `sessionid` |
| JWT в `Authorization: Bearer` | API-клиент <-> backend | подписанный токен |
| Basic Auth | служебный `/metrics` | base64(login:pass) |

Роли (если есть): `user`, `manager`, `admin`. Кратко описать, что отличает их по доступу.

**5.5.2 Ценные данные и секреты**

| Данные | Где хранятся |
|---|---|
| Учётные записи (хеш пароля) | таблица `users` в PostgreSQL |
| Сессии | Redis |
| Платёжные данные | внешнее API провайдера | 
| Логи приложения | volume `./logs` | 

**Секреты приложения** (без значений!):

- `DJANGO_SECRET_KEY` — задаётся в `.env`.
- `POSTGRES_PASSWORD` — `.env`.
- `JWT_PRIVATE_KEY` — монтируется как файл в контейнер.
- `STRIPE_API_KEY` — `.env`.
- Наблюдение: пример `.env.example` в репозитории содержит **реалистично выглядящий ключ** —
  проверить, не настоящий ли он (если да — это утечка в публичном репо).

#### 5.6 Границы доверия

| Граница | Где проходит | Что должно быть на границе |
|---|---|---|
| Клиент <-> Сервер | Браузер пользователя <-> nginx |Серверная валидация, аутентификация, rate limit, security headers |
| Сервер <-> БД | `app` <-> `db` (Docker-сеть) | Минимальные права у БД-пользователя приложения |
| Сервер <-> Внешний API | `app` <-> платёжный провайдер | Whitelist URL, mTLS/HTTPS, ротация ключей |
| Разработчик <-> Репозиторий | Локальный git <-> GitLab | Pre-commit secrets scan, code review |
