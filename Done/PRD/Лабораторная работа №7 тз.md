## 2. Теоретическая часть

### 2.1. Что такое секрет и почему утечки опасны

**Секрет** — любая информация, раскрытие которой может привести к компрометации приложения, инфраструктуры или данных.

К секретам относятся:

- API-ключи и access token;
- JWT secret (ключ подписи токенов);
- пароли и строки подключения к БД;
- SSH-ключи и TLS/SSL-сертификаты;
- cloud credentials (AWS/GCP/Azure);
- service account keys и credentials внешних сервисов;
- приватные ключи подписи (артефактов, кода).

Утечка секрета — это не «потеря пароля», а получение злоумышленником легитимного доступа. Атакующий не ломает периметр — он входит с валидными учётными данными, поэтому такая компрометация плохо детектируется средствами защиты периметра и может оставаться незамеченной месяцами.

### 2.2. Где утекают секреты: четыре точки жизненного цикла

Наиболее распространённые причины утечек:

| Точка | Типичная причина |
|---|---|
| Рабочее место разработчика | hardcoded credentials в коде, `.env` в рабочей директории |
| Система контроля версий | коммит секрета, публикация `.env`, коммит тестовых ключей |
| CI/CD-пайплайн | секреты в логах сборки, в переменных без маскирования, в артефактах |
| Среда выполнения | один секрет на все окружения, отсутствие ротации |

Ключевое наблюдение: секрет, единожды попавший в Git, **остаётся в истории навсегда**, даже если файл удалён следующим коммитом. Поэтому дешевле всего перехватить утечку *до* коммита.

### 2.3. Принцип эшелонированной защиты (defense in depth)

Один уровень контроля недостаточен. Локальная проверка (pre-commit) **обходится тривиально**:

- разработчик отключил хук (`git commit --no-verify`);
- коммит сделан через web-интерфейс GitLab;
- push с другого устройства, где хук не установлен;
- слияние чужой ветки, в которой проверки не было.

Поэтому в работе обязательны **два независимых уровня контроля**:

1. **Локальный** — pre-commit hooks (быстрая обратная связь разработчику, дешёвый перехват).
2. **Серверный** — сканирование в GitLab CI/CD (нельзя обойти, работает для всех способов внесения изменений).

### 2.4. Локальный контроль: pre-commit hooks

`pre-commit` — фреймворк управления Git-хуками. Хук `pre-commit` запускается **перед** фиксацией коммита и, если проверка вернула ненулевой код возврата, **отменяет** `git commit`.

Рабочий процесс:

1. Установка фреймворка: `pip install pre-commit`.
2. Конфигурация в `.pre-commit-config.yaml` (какие проверки запускать).
3. Установка хуков в репозиторий: `pre-commit install`.
4. Далее каждый `git commit` автоматически прогоняет secret-scanner по изменяемым файлам.

Доступные secret-scanning хуки перечислены на <https://pre-commit.com/hooks.html> (например, `detect-secrets`, `gitleaks`, `trufflehog`, `git-secrets`).

> Если что хуки есть не только в рамках сканирования секретов, также форматирования, линтер кода  и тому подобное. Полезно при собственной разработке

### 2.5. Как работают секрет-сканеры: сигнатуры, регулярные выражения, энтропия

Секрет-сканеры используют две основные техники обнаружения:

- **Сигнатурный/regex-поиск** — поиск по известным форматам ключей. Например, токен GitHub имеет префикс `ghp_` и фиксированную длину; ключ Stripe начинается с `sk_live_`. Точно, мало ложных срабатываний, но ловит только *известные* форматы.
- **Энтропийный анализ** — поиск строк с высокой случайностью (например, base64/hex-строка длиной 40+ символов). Ловит «неизвестные» секреты, но даёт больше ложных срабатываний (хеши, UUID, минифицированный код).

**Ограничение стандартных сигнатур:** они не знают специфики конкретного проекта. Если приложение использует `TELEGRAM_BOT_TOKEN`, `JWT_SECRET` или внутренний формат ключа, базовый набор сигнатур может его пропустить. Поэтому необходимо добавлять **пользовательские правила** — на основе регулярных выражений, имён переменных или формата ключей.

Примеры паттернов, которые имеет смысл описать правилами:

```text
TELEGRAM_BOT_TOKEN=
JWT_SECRET=
postgres://
-----BEGIN PRIVATE KEY-----
```

### 2.6. Контроль в CI/CD и режим Security Gate

В пайплайне сканирование выносится в отдельный job на ранней стадии (`pre-build`). Он должен запускаться при создании и обновлении `Merge Request` и при `push` в default-ветку — то есть до того, как изменения попадут в основную ветку или будут собраны.

**Security Gate** — режим, при котором обнаружение секрета *останавливает весь пайплайн*:

- джоба завершается с ошибкой (ненулевой exit code);
- сборка, тесты и деплой не выполняются;
- результат сканирования сохраняется как артефакт для разбора.

Это превращает проверку из «информационной» в «блокирующую»: небезопасный код физически не может быть собран и задеплоен. Принципиально, чтобы у job **не было** `allow_failure: true` — иначе gate не работает.

### 2.7. Проблема истории Git

Важнейший практический момент. Если секрет закоммичен, а потом удалён обычным коммитом — он **остаётся в истории** и доступен через `git log`, `git show`, в любом форке и в каждой локальной копии репозитория.

Из этого следуют два правила:

1. **Любой закоммиченный секрет считается скомпрометированным.** Первое действие при утечке — *ротация* (отзыв старого секрета и выпуск нового), а не «удаление файла».
2. Очистка истории (`git filter-repo`, BFG) — отдельная сложная операция, переписывающая историю; она *не отменяет* необходимость ротации.

Сканер в CI обычно проверяет историю коммитов, а не только текущее состояние файлов — поэтому он находит и «старые» утечки.

### 2.8. Централизованное хранилище секретов: HashiCorp Vault

Перехват утечек — это защита. Но корень проблемы — *секреты лежат не там, где должны*. Правильное решение: секрет вообще не должен находиться ни в репозитории, ни в конфигурационном файле. Он должен храниться в специализированном хранилище и выдаваться приложению в момент запуска.

**HashiCorp Vault** — централизованное хранилище секретов. Ключевые понятия:

- **Secrets engine** — модуль хранения. Для статических секретов используется **KV (Key-Value)**, версия 2 поддерживает версионирование.
- **Auth method** — способ аутентификации клиента (token, AppRole, Kubernetes и др.). Для приложений в CI/CD типичен **AppRole**: приложение предъявляет `role_id` + `secret_id` и получает токен.
- **Policy** — политика доступа: какой клиент к каким путям имеет доступ (least privilege).
- **Dev mode** (`vault server -dev`) — учебный режим: хранилище в памяти, KV v2 включён на пути `secret/`, известный root-токен. **Для production непригоден** (данные не персистентны, нет уплотнения, всё разблокировано).

Структура путей строится иерархично — по приложению и окружению:

```text
secret/myapp/dev/db
secret/myapp/dev/api
secret/myapp/prod/payment
```

#### Способы получения секретов приложением из Vault

**Базовый принцип.** Секрет не должен находиться в репозитории или в конфигурационном файле, который коммитится. Он попадает к приложению **в момент запуска или работы** — из Vault. Различаются способы тем, *кто* обращается к Vault (само приложение или внешний компонент) и *где* в итоге оказывается значение секрета (в памяти процесса, в переменной окружения или в файле на диске). От этого напрямую зависит уровень защищённости.

##### Способ 1. Через SDK / API (приложение обращается к Vault само)

Приложение в своём коде аутентифицируется в Vault и читает нужные пути через официальный клиент (`hvac` для Python, `vault-client` для Go/Java и т. д.) или напрямую через HTTP API.

```python
import hvac, os

client = hvac.Client(url=os.environ["VAULT_ADDR"])
# Аутентификация по AppRole — role_id/secret_id выдаются приложению, не лежат в Git
client.auth.approle.login(
    role_id=os.environ["VAULT_ROLE_ID"],
    secret_id=os.environ["VAULT_SECRET_ID"],
)
data = client.secrets.kv.v2.read_secret_version(path="myapp/dev/db")["data"]["data"]
DB_USER, DB_PASS = data["username"], data["password"]
```

- **Где живёт секрет:** только в памяти процесса. На диск и в переменные окружения не попадает.
- **Когда применять:** когда нужна ротация «на лету» и динамические секреты; когда команда готова поддерживать интеграцию в коде.

##### Способ 2. Init-скрипт при старте (секреты подтягиваются перед запуском)

Перед запуском основного процесса выполняется скрипт-обёртка: он читает секреты из Vault, формирует конфигурацию (env-переменные или конфиг-файл) и передаёт управление приложению (`exec`). Само приложение про Vault не знает.

```sh
#!/bin/sh
# entrypoint.sh — выполняется до старта приложения
export DB_PASSWORD="$(vault kv get -field=password secret/myapp/dev/db)"
exec python -m myapp        # exec заменяет процесс — скрипт не остаётся «висеть»
```

- **Где живёт секрет:** в переменных окружения или во временном файле процесса.
- **Когда применять:** статические секреты, нечастая ротация, нежелание менять код приложения.

##### Способ 3. Sidecar-контейнер (отдельный компонент обслуживает приложение)

Рядом с приложением запускается вспомогательный контейнер — обычно **Vault Agent**. Он аутентифицируется в Vault, читает секреты, рендерит их в файлы на общем томе и автоматически продлевает токен/обновляет значения. Приложение читает готовый файл. Типично для Kubernetes (Vault Agent Injector через аннотации).

```yaml
# Аннотации Pod в Kubernetes — Injector сам добавит sidecar Vault Agent
annotations:
  vault.hashicorp.com/agent-inject: "true"
  vault.hashicorp.com/role: "myapp"
  vault.hashicorp.com/agent-inject-secret-db: "secret/myapp/dev/db"
# Результат окажется в файле /vault/secrets/db внутри Pod
```

- **Где живёт секрет:** в файле на общем томе (желательно `tmpfs`/в памяти, не на постоянном диске).
- **Когда применять:** контейнерные/Kubernetes-окружения, много сервисов, нужна автоматическая ротация без правок кода.

##### Способ 4. Environment variables, формируемые при запуске

Вариант, близкий к init-скрипту, но секреты подставляются в процесс как переменные окружения специальной утилитой-обёрткой (`vault-env`, `envconsul`, `vault agent exec`). Приложение читает `os.environ` и про Vault не знает.

```sh
# Утилита получает секреты из Vault и запускает приложение с ними в окружении
vault-env -- python -m myapp
```

- **Где живёт секрет:** в переменных окружения процесса.
- **Когда применять:** быстрое внедрение, статические секреты, окружение, где риск чтения `/proc` приемлем.




### 2.9. Разделение секретов по окружениям и ротация

- **Разные секреты для разных окружений.** Один общий секрет для dev/staging/prod означает, что компрометация тестового стенда = компрометация production. Пути `secret/myapp/dev/...` и `secret/myapp/prod/...` разделяют их и физически, и по политикам доступа.
- **Ротация.** Секрет должен иметь срок жизни и регулярно меняться; централизованное хранилище делает ротацию выполнимой — секрет меняется в одном месте, а не «во всех конфигах».
- **Аудит.** Vault логирует обращения к секретам — видно, кто и когда читал значение.

## 4.1. Базовое задание (2 балла)

### Часть 1. Настройка pre-commit для поиска секретов

**Цель части.** Реализовать защиту репозитория от случайного добавления секрета в коммит. Проверка должна срабатывать на машине разработчика **до** выполнения `git commit` и отменять коммит при обнаружении секрета.

#### Порядок выполнения

1. **Установить фреймворк pre-commit** и подключить его к репозиторию:

   ```bash
   pip install pre-commit
   cd <ваш-репозиторий>
   ```

2. **Выбрать secret-scanning хук** из списка <https://pre-commit.com/hooks.html>. Допускаются `detect-secrets`, `gitleaks`, `trufflehog`, `git-secrets`, custom regex scanner. **Рекомендуется `gitleaks`** — тот же инструмент используется в Части 2, что даёт единые правила на обоих уровнях.

3. **Создать `.pre-commit-config.yaml`** в корне репозитория:

   ```yaml
   # .pre-commit-config.yaml
   repos:
     - repo: https://github.com/gitleaks/gitleaks
       rev: v8.18.4   # закрепите актуальную версию: pre-commit autoupdate
       hooks:
         - id: gitleaks
   ```

4. **Установить хуки в репозиторий:**

   ```bash
   pre-commit install
   ```

   После этой команды каждый `git commit` автоматически прогоняет сканер по изменяемым файлам. Без `pre-commit install` конфиг есть, но проверки **не запускаются** — это типичная ошибка.

5. **Реализовать пользовательские правила.** Стандартные сигнатуры не знают специфики проекта. Проанализируйте стек технологий и определите, какие секреты вероятны именно здесь. Реализуйте **не менее трёх** правил — на основе регулярных выражений, имён переменных или формата ключей. Для `gitleaks` правила задаются в `.gitleaks.toml`:

   ```toml
   # .gitleaks.toml — дополняет стандартные правила gitleaks
   [extend]
   useDefault = true

   [[rules]]
   id = "telegram-bot-token"
   description = "Telegram Bot Token"
   regex = '''(?i)(telegram[_-]?bot[_-]?token)\s*[:=]\s*['"]?\d{8,10}:[A-Za-z0-9_-]{35}'''
   keywords = ["telegram_bot_token", "tg_bot_token"]

   [[rules]]
   id = "app-jwt-secret"
   description = "JWT signing secret in app config"
   regex = '''(?i)(jwt[_-]?secret)\s*[:=]\s*['"]?[A-Za-z0-9+/=_-]{16,}'''
   keywords = ["jwt_secret"]

   [[rules]]
   id = "postgres-connection-string"
   description = "PostgreSQL connection string with credentials"
   regex = '''postgres(ql)?:\/\/[^:\s]+:[^@\s]+@'''
   keywords = ["postgres://", "postgresql://"]
   ```

6. **Продемонстрировать срабатывание контроля** на цикле «секрет найден -> коммит заблокирован -> секрет удалён -> коммит разрешён».

#### Проверка результата

Добавьте в файл строку с тестовым секретом и попробуйте закоммитить:

```bash
echo 'JWT_SECRET=s3cr3t_test_value_1234567890' >> app/config.py
git add app/config.py
git commit -m "test secret"
# Ожидается: хук падает, commit ОТМЕНЁН
```

Затем уберите секрет и повторите — коммит должен пройти. Должна быть наглядно видна последовательность:

```text
секрет найден  -> commit заблокирован
секрет удалён  -> commit разрешён
```

### Часть 2. Сканирование секретов в пайплайне

**Цель части.** Реализовать независимую от разработчика проверку в GitLab CI/CD. Локальный хук обходится (отключён, web-интерфейс, push с другого устройства, merge чужой ветки), поэтому нужен серверный уровень — его обойти нельзя. Проверка должна работать в режиме **Security Gate** и фиксировать находки в DefectDojo.

#### Порядок выполнения

1. **Выбрать инструмент.** В учебной инфраструктуре предоставлен образ `localhost:5000/gitleaks:latest` (рекомендуется gitleaks). Альтернативы: `kingfisher`, `trufflehog`, `talisman`.

2. **Добавить job сканирования** в `.gitlab-ci.yml` на стадии `pre-build`. Job должен запускаться при создании и обновлении Merge Request и при push в default-ветку.

3. **Включить режим Security Gate:** при обнаружении секрета job завершается с ошибкой, и весь pipeline останавливается — сборка и деплой не выполняются. У job **не должно быть** `allow_failure: true`.

4. **Сохранять отчёт как артефакт** со сроком хранения не менее одной недели.

5. **На стадии `.post`** автоматически импортировать отчёт в DefectDojo, связав импорт с `DEFECTDOJO_ENGAGEMENT_ID`.

#### Пример `.gitlab-ci.yml`

```yaml
stages:
  - pre-build
  - build
  - test
  - deploy

secret-scan:
  stage: pre-build
  image: localhost:5000/gitleaks:latest
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
  script:
    # --exit-code 1 -> ненулевой код возврата при находке => job падает => gate
    # --redact      -> значения секретов в отчёте маскируются
    - gitleaks detect --source . --redact
        --report-format json --report-path gitleaks-report.json
        --exit-code 1
  artifacts:
    when: always          # артефакт сохраняется даже при падении job
    expire_in: 1 week
    paths:
      - gitleaks-report.json

defectdojo-import:
  stage: .post
  image: localhost:5000/curl:latest   # образ с curl; см. примечание
  when: always            # импортировать находки даже если gate сработал
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
  script:
    - >
      curl -sS -X POST "$DEFECTDOJO_URL/api/v2/import-scan/"
      -H "Authorization: Token $DEFECTDOJO_API_KEY"
      -F "scan_type=Gitleaks Scan"
      -F "engagement=$DEFECTDOJO_ENGAGEMENT_ID"
      -F "file=@gitleaks-report.json"
```

#### CI/CD-переменные проекта

Settings -> CI/CD -> Variables, для всех отметить *Masked*:

| Переменная | Значение |
|---|---|
| `DEFECTDOJO_URL` | `http://10.0.0.20:8080` |
| `DEFECTDOJO_API_KEY` | API-токен пользователя DefectDojo |
| `DEFECTDOJO_ENGAGEMENT_ID` | ID Engagement, в который импортируются находки |

### 4.2. Дополнительное задание (2 балла)

#### Часть 3. Развёртывание Vault и перенос секретов

**Цель части:** отказаться от хранения секретов в репозитории и конфигурационных файлах, перенести их в HashiCorp Vault и обеспечить получение секретов приложением из Vault.

**Шаги:**

1. Развернуть Vault. Допустимые варианты: локально, в Docker, на ВМ. Для учебных целей разрешён **dev mode**.
2. Зафиксировать: адрес Vault, способ аутентификации, структуру путей хранения секретов.
3. Перенести все найденные/созданные секреты: удалить их из репозитория и конфигурационных файлов, сохранить в Vault.
4. Настроить получение секретов приложением из Vault (SDK/API, init-скрипт, sidecar или environment variables, формируемые при запуске).
5. Подтвердить работоспособность приложения с секретами из Vault.

**Пример развёртывания Vault в Docker (dev mode):**

```bash
docker run -d --name vault --cap-add=IPC_LOCK \
  -e 'VAULT_DEV_ROOT_TOKEN_ID=root-token' \
  -e 'VAULT_DEV_LISTEN_ADDRESS=0.0.0.0:8200' \
  -p 8200:8200 hashicorp/vault:latest
```

```bash
export VAULT_ADDR='http://127.0.0.1:8200'
export VAULT_TOKEN='root-token'
vault status
```

**Структура путей** (по приложению и окружению):

```text
secret/myapp/dev/db
secret/myapp/dev/api
secret/myapp/prod/payment
```

**Создание секретов** (значения не раскрывать — показать команды с маскированием):

```bash
vault kv put secret/myapp/dev/db   username=*** password=***
vault kv put secret/myapp/dev/api  api_key=***
```

**Получение секрета приложением (пример через API/SDK):**

```python
# Чтение секрета приложением при запуске (hvac — официальный Python-клиент Vault)
import hvac, os

client = hvac.Client(url=os.environ["VAULT_ADDR"], token=os.environ["VAULT_TOKEN"])
secret = client.secrets.kv.v2.read_secret_version(path="myapp/dev/db")
DB_USER = secret["data"]["data"]["username"]
DB_PASS = secret["data"]["data"]["password"]
# DB_USER / DB_PASS используются в рантайме и НЕ записываются в файл репозитория
```

> Запрещается возвращать секреты обратно в Git-репозиторий. Доступ приложения должен быть ограничен политикой (least privilege): приложение читает только свои пути `secret/myapp/<env>/...`.

## 5. Требования к отчёту


**Структура отчёта:**

- Титульный лист
- Оглавление
- **Часть 1. Настройка pre-commit для поиска секретов**
  - выбранный хук, конфигурация `.pre-commit-config.yaml`;
  - пользовательские правила (не менее трёх);
  - демонстрация блокировки и исправления (до/после).
- **Часть 2. Сканирование секретов в пайплайне**
  - выбранный инструмент и обоснование;
  - правила запуска job (MR + default branch, стадия `pre-build`);
  - отчёты/артефакты, демонстрация Security Gate (падение pipeline при секрете);
  - отправка отчёта в DefectDojo и подтверждение findings в Engagement.
- **Часть 3. Развёртывание Vault и перенос секретов** *(дополнительное задание)*
  - способ развёртывания и адрес Vault;
  - структура путей хранения;
  - способ аутентификации приложения;
  - подтверждение работоспособности приложения.
- **Итог: устранение утечек и повторная проверка**
  - результаты повторных прогонов pre-commit и CI (секретов не обнаружено).
